## 一、先感受一下：从 HTML 到视频有多简单

Hyperframes 的核心理念可以用一行命令体验：

    npx hyperframes init my-video
    cd my-video
    npx hyperframes preview      # 浏览器实时预览
    npx hyperframes render       # 渲染成 MP4

你写的"视频脚本"，就是一个普通的 HTML 文件：

    <div id="stage" data-composition-id="my-video"
         data-start="0" data-width="1920" data-height="1080">  \<video data-start="0" data-duration="5"
             data-track-index="0" src="bg.mp4" muted playsinline\>
    </video>  \<img class="clip" data-start="2" data-duration="3"
           data-track-index="1" src="logo.png" /\>
      \<audio data-start="0" data-duration="9"
             data-track-index="2" data-volume="0.5" src="music.wav"\>
    </audio>\</div\>

看到没？就是标准的 HTML 标签 + `data-*` 属性定义时间轴。

* • `data-start` = 第几秒出现
* • `data-duration` = 持续几秒
* • `data-track-index` = 图层顺序（0=最底层）

**不需要学 React，不需要装 Webpack，不需要任何构建步骤。**

你在浏览器里能看到的任何效果------CSS 动画、GSAP 时间轴、Three.js 3D、WebGL Shader------全部可以精确渲染进视频。

这就是 Hyperframes 的第一个杀手锏：**HTML-native，零构建门槛。**

## 三、核心原理：Puppeteer 怎么把网页变成视频

这是最关键的部分，���是 Hyperframes 技术壁垒最高的地方。

### 3.1 两种帧捕获模式

Hyperframes 的引擎 (`packages/engine`) 使用 Puppeteer 控制 Headless Chrome，提供两种捕获模式：

**模式一：BeginFrame API（Linux 专属）**

    // 直接操控 Chrome 的合成器，原子级控制每一帧
    const mode: CaptureMode = headlessShell && isLinux && !forceScreenshot
      ? "beginframe"
      : "screenshot";

通过 CDP（Chrome DevTools Protocol）的 `HeadlessExperimental.beginFrame` 接口，直接命令浏览器合成器："画第 N 帧"。不走渲染→截图的完整管线，而是**直接控制合成器的时钟**。

优点：极快，适合批量渲染。  
缺点：只在 Linux 上工作，且不支持透明通道（Alpha）。

**模式二：Page.captureScreenshot（全平台兜底）**

    // 先 seek 到目标时间点，再截图
    async function pageScreenshotCapture(page, time) {
      await page.evaluate((t) => window.__hf.seek(t), time);
      return await page.screenshot({ type: 'png' });
    }

这是 Remotion 也用的经典方案：seek → 等待渲染稳定 → 截图。所有平台都能跑，支持透明背景。

### 3.2 Seek 协议：让时间可以被"暂停"

HTML 页面怎么做到"精确到某一帧"？Hyperframes 在页面里注入了一个运行时协议：

    // 注入到每个 composition 页面中的协议
    window.__hf = {
      seek(timeMs) {
        // 暂停所有 GSAP timeline
        gsap.globalTimeline.pause();
        // seek 到指定时间
        for (const adapter of frameAdapters) {
          adapter.seekFrame(timeToFrame(timeMs, fps));
        }
      },
      getDuration() { /* ... */ }
    };

关键在于 **GSAP 的 seek 能力**。GSAP（GreenSock Animation Platform）是前端最强大的动画库，它的 timeline 可以精确 seek 到任意时间点并暂停在那里。

这意味着：

* • 你用 GSAP 写的 easing、stagger、timeline 全部保留
* • 引擎只需要调用 `timeline.seek(frame/fps)` 就能定位到任意帧
* • **同一份 HTML，浏览器里播放和渲染出来的视频完全一致**

这就是为什么 Hyperframes 敢说"deterministic rendering"------**相同输入，像素级一致输出。**

## 四、Frame Adapter：动画库的"万能翻译官"

Hyperframes 最精妙的设计之一是 **Frame Adapter 模式**。

问题很简单：不同的动画库有不同的时间模型。GSAP 有 timeline.seek()，CSS Animation 有 animation-play-state，Lottie 有 goToAndStop()......怎么统一？

答案是一个极简接口：

    interface FrameAdapter {
      id: string;
      init(): void;                    // 初始化
      getDurationFrames(): number;     // 总帧数
      seekFrame(frame: number): void;  // 跳转到指定帧
    }

目前内置了 4 个适配器：


<br />

<br />

<br />

<br />

|      适配器       |            支持的动画             |                     特点                     |
|----------------|------------------------------|--------------------------------------------|
| GSAP Adapter   | GSAP Timeline                | **主力**，最成熟，支持 easing/stagger/scrollTrigger |
| CSS Adapter    | CSS Animations / Transitions | 零依赖，原生支持                                   |
| Lottie Adapter | Lottie JSON (Bodymovin)      | 设计师友好，After Effects 导出                     |
| 自定义 Adapter    | 任意 JS 动画                     | 实现 FrameAdapter 接口即可                       |

<br />


<br />

GSAP 适配器的实现只有 **50 行代码**：

    export function createGSAPFrameAdapter(options): FrameAdapter {
      const { fps, timeline } = options;  return
     {
        id: options.id ?? "gsap",
        init: () => { timeline.pause?.(); },
        getDurationFrames: () => Math.ceil(timeline.duration() * fps),
        seekFrame: (frame) => {
          timeline.pause?.();
          timeline.seek(frame / fps, false);
        }
      };
    }

**50 行代码，打通了 GSAP 生态和视频渲染引擎之间的桥梁。**

![](https://cubox.pro/c/filters:no_upscale()?imageUrl=https%3A%2F%2Fmmbiz.qpic.cn%2Fsz_mmbiz_jpg%2F831yxq96lHVT1TvGLINeRUWU7zc6fjLYuNEdTBpwk8Vl4vhlDXmsV7ROSEaVr67wfFDEwBa8vQAvL7nFib3w0wKLOFeu46vM3mMVAzzdchW4%2F640%3Fwx_fmt%3Djpeg%26watermark%3D1%23imgIndex%3D2)

这意味着什么？意味着 GSAP 生态里所有的插件------ScrollTrigger、MotionPath、DrawSVG、SplitText------理论上都可以在 Hyperframes 里被逐帧渲染成视频。

这是一个巨大的护城河。因为 GSAP 有 **超过 10 年的动画积累**，而 Hyperframes 把这 10 年的成果一次性搬到了视频渲染领域。

## 五、流式编码引擎：image2pipe 管道的艺术

帧捕获完了，怎么变成视频？这里有个很酷的设计：**流式编码（Streaming Encoder）。**

传统做法是两步走：

1. 1. 截图保存到磁盘（1000 帧 = 1000 张 PNG）
2. 2. FFmpeg 读磁盘图片 → 编码成视频

问题是：**磁盘 I/O 太��了**，尤其是 4K 分辨率下每张 PNG 可能好几 MB。

Hyperframes 的方案：

    // 流式编码：帧截图直接通过 stdin 管道送给 FFmpeg
    // 不写磁盘！
    const ffmpeg = spawn('ffmpeg', [
      '-f', 'image2pipe',   // 从 stdin 读取图片流
      '-i', '-',             // stdin
      '-c:v', 'libx264',    // H.264 编码
      '-pix_fmt', 'yuv420p',
      '-r', String(fps),     // 帧率
      'output.mp4'
    ]);// 每截完一帧，直接写入管道

    function writeFrame(buffer) {
      ffmpeg.stdin.write(buffer);
    }

但这里有个技术难点：**并行捕获的帧是乱序完成的**。Worker A 可能在处理第 50 帧，Worker B 已经完成了第 60 帧。但 FFmpeg 要求帧必须按顺序输入。

解决方案是一个 **Frame Reorder Buffer（帧重排缓冲区）**：

    function createFrameReorderBuffer(startFrame, endFrame) {
      let cursor = startFrame;  // 当前期望的帧号
      const pending = new Map();  return
     {
        waitForFrame(frame) {
          if (frame === cursor) return Promise.resolve();
          return new Promise(resolve => enqueueAt(frame, resolve));
        },
        advanceTo(frame) {
          cursor = frame + 1;
          flushAt(frame);  // 唤醒等待该帧的 worker
        }
      };
    }

这是一个经典的**有序异步屏障（Ordered Async Barrier）**模式：

* • 并行 Worker 任意速度生产帧
* • 缓冲区保证按序写入 FFmpeg stdin
* • 最早到达的帧先写，后面的排队等

**效果：在 8 核机器上，渲染速度比串行快 3-5 倍。**

![](https://cubox.pro/c/filters:no_upscale()?imageUrl=https%3A%2F%2Fmmbiz.qpic.cn%2Fsz_mmbiz_jpg%2F831yxq96lHXndDlgmqkZHIBUz8vehibffLnnJIkNu72bYpW6cQKkIbEGy53cjmLcAUvoBvaSvnJLRibxPCIltFrAfcdTpzpxeKoWd5nr7aF4o%2F640%3Fwx_fmt%3Djpeg%26watermark%3D1%23imgIndex%3D3)

## 八、Hyperframes vs Remotion：一场关于"HTML or React"的战争

既然提到了 Remotion，就不得不聊聊这两个项目的关系。

Remotion 是浏览器视频渲染领域的**先行者** ，2020年就发布了，目前在 GitHub 上有 **28k+ 星**，被大量生产环境使用。Hyperframes 的 README 里明确写了：
> *We used Remotion at HeyGen in production, learned a ton from it.*

Hyperframes 的源码注释里甚至保留了 Remotion 的 attribution（致敬标注）。

但两者做了一个根本性的不同选择：


<br />

<br />

<br />

<br />

|       维度        |        **Hyperframes**         |      **Remotion**      |
|-----------------|--------------------------------|------------------------|
| **创作方式**        | HTML + CSS + GSAP              | React 组件 (TSX)         |
| **构建步骤**        | **无**，index.html 直接用           | 必须（Webpack/esbuild 打包） |
| **动画库**         | GSAP / Lottie / CSS / Three.js | React state 驱动         |
| **Seekable 动画** | ✅ GSAP 原生支持 seek               | ⚠️ 壁钟时间播放，非帧精准         |
| **HTML/CSS 透传** | ✅ 粘贴即用                         | ❌ 必须改写成 JSX            |
| **分布式渲染**       | 单机（当前）                         | Lambda 云渲染（成熟）         |
| **许可证**         | **Apache 2.0（真正开源）**           | 自定义许可证（商业限制）           |

<br />


<br />

**最大的分歧点：React 还是 HTML？**

Remotion 的哲学是：**前端已经用 React 了，视频也应该是 React 组件。** 这对 React 开发者很自然，但代价是你需要把所有东西改写成 JSX------包括现有的 HTML/CSS 代码。

Hyperframes 的哲学是：**HTML 就是最好的 DSL。** 浏览器已经花了 30 年打磨 HTML/CSS/SVG/WebGL，为什么要再抽象一层？

两种哲学都有道理。但如果你：

* • 有现成的 HTML 页面想转视频 → **Hyperframes 天然适配**
* • 用 GSAP/Lottie 做过动画 → **Hyperframes 直接兼容**
* • 需要 Apache 2.0 商用无忧 → **Hyperframes 胜出**
* • 需要 Lambda 分布式云渲染 → **Remotion 更成熟**

这不是谁替代谁的关系，而是**同一个赛道上的两种不同解法。**

## 九、谁在做这件事？Heygen 的野心

聊技术聊够了，说说背后的人。

**Heygen** 是 AI 视频赛道的头部玩家，主打 AI 数字人和视频翻译。2024 年 ARR 破亿美元，是 AI 视频领域增长最快的公司之一。

那 Heygen 为什么开源 Hyperframes？

从源码里的线索来看：

1. 1. **内部已经在用了**：README 写了 "used Remotion at HeyGen in production"，说明他们有真实的视频渲染需求------产品介绍视频、客户案例视频、营销素材，这些都需要批量自动化生产。
2. 2. **Agent 是未来** ：Hyperframes 的 Skills 系统明显是为 Claude Code / Cursor / Codex 等 AI 编程 Agent 设计的。Heygen 判断：**未来的视频制作工作流是 Agent 驱动的。**
3. 3. **建立生态标准**：通过开源 + Vercel Skills 规范，Hyperframes 试图成为 "AI Agent 视频渲染" 事实标准。当所有 Agent 都学会用 Hyperframes 做 video 时，Heygen 就站在了这个生态的入口。
4. 4. **Apache 2.0 许可证** ：完全开源，商用无限制。这与 Remotion 的自定义许可证形成鲜明对比。Heygen 不靠卖许可证赚钱，而是靠**生态位**赢。

从 contributors 来看，核心团队主要是 Heygen 内部工程师：

## 十、为什么这件事比你想的更重要

让我们跳出技术细节，看看更大的图景。

**视频正在成为新的"文本"。**

以前我们用 Markdown 写文档，用 HTML 做网页。现在呢？抖音是视频，小红书是图文，YouTube 是长视频，TikTok 是短视频。信息消费正在全面视频化。

但视频制作的门槛依然很高。你需要懂 Premiere，懂 After Effects，懂调色，懂节奏。**一个工程师可能写出世界级的代码，但做一个 30 秒的产品介绍视频都要折腾半天。**

Hyperframes 做的事情，本质上是**把视频制作的门槛降到了"会写 HTML"的程度。**

而结合 AI Agent，这个门槛进一步降到了**"会用自然语言描述"的程度。**

你说"帮我做一个 10 秒的产品介绍视频"，Agent 写 HTML，Hyperframes 渲染成 MP4。

这不是科幻。这是今天你已经能做的事情。
