# Vercel 设计团队，已经不按传统设计流程干活了

**Vercel 设计团队到底在用什么工具，答案可能和你想的不一样**

他们没有整齐划一地迁移到某一个 AI 工具，也没有抛弃 Figma。真正发生的，是设计、代码、浏览器和 agent 正在缠成一套新的混合工作流。


<br />

<br />


<br />

**这篇文章最有意思的，不是一串工具名单，而是让人看见一个顶级团队此刻真实的工作现场。**

那里没有统一答案，只有不同的人、不同的习惯、不同的工具，以及一套还在快速变形的新流程。


<br />

<br />


<br />

这两天我看到 Hannah Hearth 发的一条推文，挺值得展开聊聊。

她说，现在关于设计工具的讨论，已经快被各种大而空的判断淹没了。于是她写了一篇更具体的长文，直接讲 Vercel 产品设计团队当下到底在用什么工具。

这个角度很好。因为现在大家一聊设计工具，最常听见的几句话都差不多：Figma 要被替代了，AI 会重写设计流程了，设计师都该学代码了，未来没人再画原型了。听着热闹，但很多时候还是太虚。

<br />

![](https://cubox.pro/c/filters:no_upscale()?imageUrl=https%3A%2F%2Fmmbiz.qpic.cn%2Fmmbiz_jpg%2FSrpPXmsQ1BSTemBqs1bP8YXKYwH9aK770f55qZ2xD3yoEN1jW0RKd7E13fbbb0G5SqgUH4ZYWDkwmibewodBSQAia9BShicgvibNBV1ae0kB59Q%2F640%3Ffrom%3Dappmsg%26watermark%3D1%23imgIndex%3D0)

<br />

真正值得看的，从来不是谁喊得最猛，而是顶级团队今天到底怎么干活。他们遇到什么问题，用什么工具，哪些流程真的变了，哪些其实还没变。Hannah 这篇文章最有价值的地方，就在它给了很多非常具体的现场切片。

<br />

### 同一个团队里，大家已经活在不同世界

<br />

Hannah 一上来就提到一个很真实的现象：哪怕在 Vercel 这样规模不算特别大的产品设计团队里，每个人现在用的工具都完全不一样。

这句话信息量很大。很多人想象中的 AI 设计团队，应该会很快收敛到一套新范式，比如统一用某个 AI 产品、某种 agent 工作流、某套代码驱动设计流程。现实显然没这么整齐。

在同一个团队里，有人已经六个月没有一天离开 AI，也有人依然觉得 AI 在创意工作里大多数时候都不太好用。这说明设计行业现在还处在很早期的过渡阶段。工程侧已经逐渐出现一些稳定共识，设计侧还远远没有。

<br />


<br />

**更准确地说：**   
设计团队现在不是在同一条新跑道上提速，而像是一群人同时在试不同的桥，看看哪座不会塌。


<br />

<br />

### AI 设计工具现在最大的问题，是不够顺

<br />

文章里有一句判断很准，设计工作流里的 AI 工具，整体上比工程工作流里的 AI 工具更笨、更卡、更不顺手。

这其实解释了很多现象。为什么不少设计师尝试了一圈 AI-first 工作流，最后又回到 Figma？不是他们保守，也不一定是抗拒新工具，而是很多 AI 设计工具现在还不够顺。

设计工作非常依赖连续性。你在画布里快速发散、快速试、快速比较，脑子和手是连着的。这个节奏一旦被打断，人就容易掉出状态。现在很多 AI 工具的使用方式，偏偏是"发个 prompt，然后等"。等着等着，人就开始刷 Slack、回消息、切别的事，注意力断了，流也断了。

<br />

![](https://cubox.pro/c/filters:no_upscale()?imageUrl=https%3A%2F%2Fmmbiz.qpic.cn%2Fsz_mmbiz_jpg%2FSrpPXmsQ1BQtAMqCuk6CGMyY0ep9feQG6eK0JmiaRMNiabTCmEy36LsBgp9K7y2CSVbYeib4HIibDKW7ibxdw1BeWTElmibeiaIEJicy6N9xARftW24%2F640%3Ffrom%3Dappmsg%26watermark%3D1%23imgIndex%3D1)

<br />

这不是体验上的小瑕疵，而是在动创作节奏本身。AI 提升了执行能力，却也打断了很多人最宝贵的专注状态。所以 Hannah 文里提到一个很重要的变化：想进入 flow state，就得学会并发线程。

<br />

### 未来的设计师，可能越来越像 agent 调度员

<br />

Vercel 团队里已经有人这么干了。开一个项目，同时开多个标签页、多个线程，把问题拆开跑。一个线程处理 UX flow，一个线程处理 state 和 data，另一个线程专门修零碎的小问题。

这个做法和工程师已经熟悉的多 agent 协作很像，只是它现在被搬进了设计工作里。过去我们理解设计师的核心能力，通常是审美、交互判断、信息组织、用户理解，再加上对工具的熟练度。接下来，这份能力清单里很可能要多出一项：任务编排。

谁更会拆问题，谁更会并发推进，谁更会让多个 agent 分头工作还不失控，谁就更容易在新的工具链里保持速度。这时候，设计师的角色会开始变化。他不只是亲手画图的人，也像是在带一群数字实习生干活的人。

<br />

### 设计稿正在失去"领先于生产"的位置

<br />

整篇文章里，我最在意的是这一段。Hannah 提到，在旧世界里，设计文件总是领先于生产环境，团队会努力让设计稿和线上产品保持同步。现在这件事正在失效。

原因也直接。实现成本和实现速度都降得太快了，很多功能直接就在代码里长出来了，设计稿反而容易变旧。线上产品已经走到了前面，画布上还停在前几步。

这意味着，生产环境正在慢慢变成新的事实来源。以前设计稿是源头，代码是落地。现在越来越多场景里，代码和生产环境反而跑到了前面，设计稿退成了一个辅助性的探索工具。

<br />

这也是为什么 Vercel 团队开始研究怎么把生产环境重新拖回画布里。有人用 Paper 把页面结构和样式带回设计画布，有人用自定义 Chrome 插件快速跳到组件文件位置，还有人直接在 Slack 里 @ agent 改代码、开 PR，连 backlog 都懒得先记一笔。

他们没有死守旧流程，而是在接受一个现实：既然生产环境已经成了新的中心，设计工具也要围着它重建。

<br />

### 未来有一类设计迭代，可能根本不需要传统画布

<br />

文章里还有一个很亮的点，是一个叫 UI Fork 的内部工具。它可以在本地开发环境里生成同一个组件的多个变体，让你直接在浏览器里切换、比较、评审，最后把选中的版本回写进原文件。

这个思路很重要。它解决的是 AI coding 工具做设计时最难受的一个问题：没有画布，你很容易一路钻进一个单线版本，缺少横向发散和并排比较。UI Fork 相当于在浏览器里造了一个"变体画布"。

这件事背后的信号很清楚。下一代设计画布，不一定长得像 Figma。它可能根本不是传统意义上的画布工具，而是浏览器、本地环境、组件系统和 agent workflow 拼出来的一种新形态。

<br />

### Figma 还没出局，但它的位置已经变了

<br />

这篇文章并没有简单唱衰 Figma，反而很诚实。Vercel 团队里仍然有人觉得，Figma 在大范围探索时更自然。也有人试了一圈 AI-first 设计流程，最后还是回到 Figma。

这很正常。因为 Figma 现在最强的地方，依然是"广度探索"。它仍然最适合快速拉开空间、并排试想法、保留模糊性。

但问题也很明显。和 AI coding 工具相比，Figma 现在显得慢了。Hannah 提到，她用 Figma MCP 给设计稿批量填真实数据、logo 和内容，省下了不少机械劳动。甚至会让人忍不住想一句：都 2026 年了，为什么这些事还不是原生能力？

<br />


<br />

**真正值得注意的，不是 Figma 会不会马上被替代，而是它正在被重新分配职责。**

* 它仍然适合做广度探索。
* 但它不再天然占据整条设计链路的中心
* 越来越多执行、迭代、比较、修补，已经转移到代码和浏览器里。
* 设计工具的边界，开始被重新画线

<br />


<br />

<br />

### 最后最值得记住的，其实不是哪款 AI 工具

<br />

文章最后还有一个很妙的地方。Vercel 团队分享的，不全是 AI 工具。比如 Timeless，会持续录屏，让你能随时截取刚刚过去的五分钟，专门抓那些难复现的界面 bug。比如 Cleanshot 的 Pin 工具，可以把截图叠到新设计上，快速做 before/after 对比。

这提醒我们，真正改变工作流的，不一定是最"AI"的工具。很多时候，是那些老老实实把一个具体痛点直接解决掉的小工具。

所以别把视线全盯在"哪个模型最强""哪个 agent 最聪明"上。真正有价值的问题更朴素：它有没有帮你少走一步？有没有让你更快比较版本？有没有让一个抱怨直接在 Slack 线程里变成 PR？

Hannah 这篇文章最有意思的地方，也正在这里。它没有给出一个宏大结论，却让我们看见，最好的设计团队现在也没有统一答案。大家还在边做边试，边试边改。而新的设计工作流，可能就会从这些不整齐、甚至有点混乱的尝试里长出来。
