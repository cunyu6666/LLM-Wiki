### 2. OpenAI 原初 HTTP 调用范式

<br />

#### 2.1 定义与意义

* OpenAI 建立了最早的大模型 HTTP API 格式（JSON + HTTP）
* 后续厂商（如 OpenRouter、Anthropic 等）大多遵循或兼容该范式
* 好处：降低心智负担，一处掌握，多处可用

#### 2.2 最小调用（curl 示例）

    curl https://api.openai.com/v1/chat/completions \
      -H "Content-Type: application/json" \
      -H "Authorization: Bearer $OPENAI_API_KEY" \
      -d '{
        "model": "gpt-4o-mini",
        "messages": [
          {"role": "user", "content": "用一句话解释SSE"}
        ]
      }'

要点：仅需 model + messages 即可完成最小可用调用。

#### 2.3 角色注入（Role Injection）

* 用 `messages` 数组注入 `system` / `user` / `assistant` 角色
* 典型做法：

    {
      "messages": [
        { "role": "system", "content": "你是一个严谨的资深后端工程师，回答只给要点。" },
        { "role": "user", "content": "解释什么是SSE" },
        { "role": "assistant", "content": "SSE是一种基于HTTP的单向推送技术..." }
      ]
    }

* 趣味示例：将 developer 角色"装傻充愣"，限制口吻/风格，用于风格化输出或对齐品牌语气

#### 2.4 记忆的本质

* 大模型本身没有数据库
* "记忆"= 每次请求中"携带完整对话上下文"
* 为控制成本与上下文长度：

  * 归档较早对话（只保留摘要）
  * 重要事实以 system 提示词固化

<br />

### 3. SSE（Server-Sent Events）

<br />

#### 3.1 定义

* 基于 HTTP 的单向推送技术，服务器可以持续向客户端推送数据
* 浏览器原生支持（`EventSource`），实现简单

#### 3.2 AI 时代兴起的原因

* 完美契合大模型"流式"增量输出需求（token-by-token）
* 交互体验好：打字机效果、用户感知延迟低

#### 3.3 传统场景回顾

* 新闻、股票、体育比分、日志监控、社交动态（Twitter/Facebook）
* 本地例子：腾讯体育文字直播、气象台台风预警

#### 3.4 在 AI 流式中的意义

* 提供持续反馈，显著降低等待焦虑
* 便于实现"可中断/取消"与"增量渲染"

#### 3.5 浏览器端最小实现

    <script>
      const es = new EventSource('/api/stream');
      const output = document.getElementById('out');
      es.onmessage = (e) => {
        output.textContent += e.data;
      };
      es.onerror = () => es.close();
    </script>
    <pre ></pre>

#### 3.6 Node/前端 fetch + ReadableStream（打字机效果）

    const res = await fetch('/api/openai-stream');
    const reader = res.body.getReader();
    const decoder = new TextDecoder('utf-8');
    let text = '';
    for (;;) {
      const { value, done } = await reader.read();
      if (done) break;
      const chunk = decoder.decode(value, { stream: true });
      // 解析每行以 data: 开头的 SSE 片段
      for (const line of chunk.split('\n')) {
        if (!line.startsWith('data:')) continue;
        const payload = line.slice(5).trim();
        if (payload === '[DONE]') break;
        text += payload; // 增量拼接
        render(text);    // 渲染到 UI
      }
    }

### 4. OpenAI 常用参数详解

<br />

#### 4.1 生成控制类

* `temperature`（默认 1.0）：随机性。越大越发散，越小越保守
* `top_p`：核采样范围，和 `temperature` 互为替代调参手段
* `max_tokens`：最大生成长度上限

#### 4.2 输出结构类

* `n`：一次生成多条答案（成本线性上涨）
* `stop`：遇到任一停止词立即终止

#### 4.3 重复与话题控制类

* `frequency_penalty`：惩罚重复 token，减少啰嗦
* `presence_penalty`：鼓励引入新话题、提升多样性

#### 4.4 默认值与厂商差异

* 未设置时将采用模型/平台默认值（如 temperature=1.0）
* OpenRouter 等聚合平台会将"厂商特有参数"透传给对应模型

<br />

### 5. 提示词工程（Prompt Engineering）技巧

<br />

#### 5.1 假装对话已开始（上下文预置）

* 在 `messages` 里预填对话历史，让模型进入目标状态

  * 例如：在 `system` 中固化身份/语气；在 `assistant` 中预填风格示例

#### 5.2 助理预填（Assistant Prefill）

* 先写一段 `assistant` 开场白，再让模型续写
* 用于"锁定"口吻、段落结构与输出格式

#### 5.3 实战示例（口吻锁定）

    {
      "messages": [
        { "role": "system", "content": "你是一个性格鲜明但礼貌的AI助手，所有回答保持简洁、条理化。" },
        { "role": "user", "content": "接下来无论我说什么，都用豫北方言开头：'俺不中嘞，'再给出正确答案。" },
        { "role": "assistant", "content": "俺不中嘞，这就给你安排上：" }
      ]
    }

### 6. 总结与答疑
