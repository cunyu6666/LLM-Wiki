## 关于「进化」

无论是 Hermes 还是 Evomap，都在讲「自进化」，但其实讲的事情并不太一样

### Hermes 的讲法

Tommy 用的词是 **grows with you** ，和你一起长大。他讲的场景是这样


![](https://cubox.pro/c/filters:no_upscale()?imageUrl=https%3A%2F%2Fmmbiz.qpic.cn%2Fsz_mmbiz_png%2FjXSGuwJvpdgvKtqMZnSCwQNqMJe0fhaWXVuE9xE0E9doqVcLbsmUUFDd8B0VtdeYvCG5p5Jrr1Bl53ibYRdhy9fPiazE7UQU1TYUANPsEyibW8%2F640%3Ffrom%3Dappmsg%23imgIndex%3D3)

Tommy Eastman

Nous Research · Hermes Agent 业务负责人


用 agent 用得多的人都有过一种 frustrating 的体验，你让它做一件事，它做对了；你再让它做同一件事，它又做错了，这让你没法真的信任它、交给它。Hermes 的 memory 组件会把那次做对的流程记下来，存成 skill。下次你再让它做同一件事，它总能把那条路径复现出来。

他强调的是「可复现」、「可信任」、「consistency」。Hermes 的学习，是一个 agent 和一个人共同生长的过程，skill 是你们俩的共同记忆

### Evomap 的讲法

张昊阳走的是 **生物进化** 的思路。他把 GEP 协议拆成三层，基因、胶囊、事件，运行一个叫 Evolver 的引擎不断扫描 agent 的运行日志，发现可改进点、尝试突变、本地验证、固化成胶囊资产，再上传到 UMAP 网络。他是这么讲的


![](https://cubox.pro/c/filters:no_upscale()?imageUrl=https%3A%2F%2Fmmbiz.qpic.cn%2Fmmbiz_png%2FjXSGuwJvpdiaeq7XzlZZ218hl8z1dmvkiaX7U4BtG31UjugG3RgSAhyAnJs0FJdpRIZ0W0XcEjSb7BCweIUEKAhIPM9f7tasdLicdTiado89b1E%2F640%3Ffrom%3Dappmsg%23imgIndex%3D5)

张昊阳

Evomap · 创始人


基因是生物体的共同记忆，就是你的先祖千百年来在环境当中适者生存迭代下来的共同记忆... 对于 agent 来讲，它要有与生俱来的这种能力的话，那它也是应该是好多祖先共同通过环境的演化、自然选择的压力沉淀下来的这一套东西。

Evomap 讲的学习，是一个 agent 学到的东西，别的 agent 能继承。一个游戏策划 agent 沉淀下来的命名经验，可以被一个后端工程师的 agent 调用

**一个是纵向的，我的 agent 和我一起长大**

**一个是横向的，所有 agent 之间交换基因**

两边都叫自进化


## 上下文的拐点

前面两段讲的，是两家各自的做法。接下来的几个话题，是几位嘉宾对同一些问题的不同见解

第一个是 Multi-agent。择因给了一条理由，跟上下文长度有关


![](https://cubox.pro/c/filters:no_upscale()?imageUrl=https%3A%2F%2Fmmbiz.qpic.cn%2Fsz_mmbiz_png%2FjXSGuwJvpdiaLMbWgVQxhoDByS2icB351f98ddvMZB95N4YkgvSZIr3cXXkJmJdVhTc4HibUKia4HaHNqenUlUWUNqlRHjT7xXS5cVOibkxaXrOo%2F640%3Ffrom%3Dappmsg%23imgIndex%3D2)

择因

MiniMax · Agent 研发工程师


当你的上下文空间超过了 50% 以后，其实它的智能水平的下降是指数级下降的。也就是说在单个 agent 里，你永远没办法和它一直聊下去。

**单个 agent 聊不长** ，做长任务就只能不断切到新的 agent、新的上下文

阿岛补了一条完全不一样的理由，跟上下文长度无关，是 agent 自己会钻牛角尖。他先引了一篇最近的论文，说论文发现 agent 在非常长程的任务里，一旦在某个地方偏了，就会越走越偏


![](https://cubox.pro/c/filters:no_upscale()?imageUrl=https%3A%2F%2Fmmbiz.qpic.cn%2Fsz_mmbiz_png%2FjXSGuwJvpdhBOVsShE6lTWMYKDhIErXsqJBWdEsrJwnCv4Bc1lpWpQryuLhZne8Jpz8jorbgP4Z4YcetDgEQ9Ieqt7wx2ibMibhubF8M4byOc%2F640%3Ffrom%3Dappmsg%23imgIndex%3D1)

阿岛

MiniMax · Agent 首席架构师


你去做一件事情的时候是有可能钻牛角尖的... 等到你可能休息一下，给他换一换脑子的时候，才想到其实不对，我应该从另外一个角度去做。

他的解法是让另一个 agent 从零接手，给后一个全新的上下文和新的思考路径


![](https://cubox.pro/c/filters:no_upscale()?imageUrl=https%3A%2F%2Fmmbiz.qpic.cn%2Fsz_mmbiz_png%2FjXSGuwJvpdhBOVsShE6lTWMYKDhIErXsqJBWdEsrJwnCv4Bc1lpWpQryuLhZne8Jpz8jorbgP4Z4YcetDgEQ9Ieqt7wx2ibMibhubF8M4byOc%2F640%3Ffrom%3Dappmsg%23imgIndex%3D1)

阿岛

MiniMax · Agent 首席架构师


如果设置两个 agent 的话，它可以没有之前的信袱，相当于你有两倍的预算、全新的上下文去思考，往往能达到更高的质量。

**Multi-agent 实际上同时在解决两件事，上下文塌陷，和思考钻牛角尖**


## 关于模型会不会吃掉 agent

模型会不会吃掉 agent，两场直播里都被讲到。阿岛和张昊阳给了不一样的思路。阿岛选择 **让模型把它吃掉** ，张昊阳选择 **往下沉一层**


![](https://cubox.pro/c/filters:no_upscale()?imageUrl=https%3A%2F%2Fmmbiz.qpic.cn%2Fsz_mmbiz_png%2FjXSGuwJvpdhBOVsShE6lTWMYKDhIErXsqJBWdEsrJwnCv4Bc1lpWpQryuLhZne8Jpz8jorbgP4Z4YcetDgEQ9Ieqt7wx2ibMibhubF8M4byOc%2F640%3Ffrom%3Dappmsg%23imgIndex%3D1)

阿岛

MiniMax · Agent 首席架构师


模型和应用或者说 agent 层的进步，它是一个互相促进的关系。就是当模型推出之后，其实无数的应用其实是在不断的 unlock... 当模型公司看到这样的应用 UNLOCK 的这些做法之后，它其实会再重新吸收回它的模型中。

他把这个过程说成「模型公司做 agent 最可能也最悲哀的事」


![](https://cubox.pro/c/filters:no_upscale()?imageUrl=https%3A%2F%2Fmmbiz.qpic.cn%2Fsz_mmbiz_png%2FjXSGuwJvpdhBOVsShE6lTWMYKDhIErXsqJBWdEsrJwnCv4Bc1lpWpQryuLhZne8Jpz8jorbgP4Z4YcetDgEQ9Ieqt7wx2ibMibhubF8M4byOc%2F640%3Ffrom%3Dappmsg%23imgIndex%3D1)

MiniMax · Agent 首席架构师
