## 本文涉及资源链接放在文末〈阅读资料〉章节，本文涉及源码建议在1920宽屏下阅读。

先说一下 Spec (文末有〈术语〉章节)，在软件开发领域，"specification" 翻译为"规范"还是"规格"取决于具体语境：

规范 (规范文档、标准)，通常侧重于规则、标准和要求，强调的是约束和统一的标准。  

规格 (规格说明、技术规格)，更多用于描述产品或系统的详细功能、特性和要求，偏向于具体的描述和细节。

根据 GitHub Spec Kit 的上下文，这里的"specification"更倾向于指详细的需求说明和设计文档，因此理解成"规格"会更贴切，强调的是对系统功能和行为的具体描述。

## 简介

Spec Kit 是 GitHub 开源的一个工具包，旨在通过规格驱动开发 (Spec-Driven Development, SDD) 方法，利用 AI 助手（如 CodeX、Claude Code）来提升软件开发效率和质量。

## SDD 的必要性

建议先阅读《规格驱动开发》理论，深入了解这种"方法论"，再结合下面的实操步骤进行实践。

例如，里面讲到：
> 整个开发工作流程围绕规格作为事实的中心来源重新组织，实现规划和代码作为不断再生的输出...

------ 说明在实践时需要：循环反馈，持续进化。

又讲到：
> 三种趋势：
>
> 首先，人工智能的能力已经达到了一个阈值，自然语言规格可以可靠地生成可运行的代码。
>
> 其次，软件的复杂性仍在指数级增长。通过手动流程使数十个"服务、框架和依赖项"与"原始意图"保持一致变得越来越困难，框架可能会演变为提供人工智能优先的支持，而不是人类优先的支持。
>
> 第三，变化的速度加快了。现代产品开发要求基于用户反馈、市场条件和竞争压力进行快速迭代。手动将变更传播到文档、设计和代码中，结果是要么缓慢、谨慎的更新限制了速度，要么快速、鲁莽的变更积累了技术债务。

------ 说明采用 SDD 不仅可能，而且必要。

整个核心理念是：

* 驱动规格开发是一种结构化的流程；

* 意图驱动的开发方法，先明确'做什么/What'的规格，再确定'如何做/How'；

* 通过规格约束和组织原则，保证需求规格的质量和一致性；

* 多步骤迭代优化改进，而非依赖单次从提示生成代码；

* 采用高级 AI 模型能力来理解和处理需求规格。

## SDD 实践

下面分享一下具体的操作 (macOS or Linux 平台)。

### 准备

1. 一个能访问 GitHub 的网络，在终端中临时开启代理，避免出现 GitHub API returned 403 for https://api.github.com/repos/github/spec-kit/releases/latest 异常；

2. 一个高级 AI 助理，如 CodeX、Claude Code，推荐 API 服务商 DuckCoding、PackyCode；

3. Python 3.11+ 环境，推荐使用 uv 管理 Python 版本和包。

### 第三步：建立"宪章" /speckit.constitution

这个阶段的目标是为项目创建一套治理原则和开发指南，确保后续的规格、规划和实现都遵循这些基础规则。

#### 输入

使用 /speckit.constitution 命令创建项目的治理原则和开发指南，这些将指导后续所有开发工作。

注：这里的"开发指南"是指如何开展规格驱动开发。

````

```


```


```
/speckit.constitution 制定以代码质量、测试标准、用户体验一致性和性能要求为重点的原则

```

````

这个步骤将会创建或更新 .specify/memory/constitution.md 文件，其中包含项目的基础指南，AI 代理将在规格、规划和实施阶段参考这些指南。

SDD 的核心就是这一份 Constitution (宪章)，一组不可更改的原则，它规定了规格如何成为代码，宪章 memory/constitution.md 作为系统的架构 DNA，确保每个生成的实现都保持一致性、简单性和质量。

#### 输出

更新宪章，落地完整原则和治理规则，聚焦业务定位。

### 第四步：创建功能规格 /speckit.specify

这个阶段不涉及任何具体的技术栈选择，专注于业务需求和功能规格的定义，确保规格清晰、完整且可验证。

#### 输入

在确立了项目原则之后，现在开始创建功能规格。

使用 /speckit.specify 命令，叙述想要开发的具体需求。尽可能明确地说明想要"什么/What"和"为什么/Why"，此时不要关注技术栈，提供业务性的描述。

````

```


```


```
/prompts:speckit.specify 构建一个应用程序，能够帮助我将照片整理到不同的相册中。相册按日期分组，并且可以在主页面通过拖放操作重新排列。相册之间不会嵌套，即相册不会包含在其他相册内。在每个相册内，照片以类似瓦片的界面进行预览。

```

````

不要将第一次尝试视为最终结果，此期间是与 AI 助手澄清和提问规格细节的机会，非常重要，因为后面步骤是与具体业务无关的流程了。

在基础规格创建完成后，可以通过 /speckit.clarify 指令继续澄清 (Clarification) 在初次尝试中未能正确捕捉到的任何需求，并且可以让 AI 助手评审与验收。

#### 输出

生成规格说明书内容，输出到 specs/<###-feature-name>/ 文件中 (new file)。

注：<###-feature-name> 是一个占位符，用于表示特定功能的名称。这里的 ### 指的是功能的编号，而 feature-name 是该功能的具体名称；例如，如果功能名称是 "user-authentication"，那么输出路径可能是 specs/001-user-authentication/。

## 术语

有大量的术语、缩写和编码，Spec Kit 通过提供一个结构化的方式来组织和管理这些信息，确保在整个开发过程中保持一致性和清晰度。

|                   术语                   |                                              说明                                              |
|----------------------------------------|----------------------------------------------------------------------------------------------|
| 规格                                     | 详细描述系统功能和行为的文档，强调具体的描述和细节。                                                                   |
| 规范                                     | 侧重于规则、标准和要求，强调约束和统一的标准。                                                                      |
| SDD，Specification-Driven Development   | 规格驱动开发，一种以规格为中心的开发方法论。                                                                       |
| LLM，Large Language Model               | 大型语言模型，如 GPT-4、Claude Code 等。                                                                |
| AI Assistant (Artificial Intelligence) | 基于 LLM 的智能助手，用于辅助软件开发，如 CodeX、Claude Code 等。                                                 |
| Constitution                           | 宪章，指项目的基础原则和治理规则。                                                                            |
| Gate                                   | 门禁，指在开发过程中，为了确保代码质量和规范性，设置的一系列检查机制，如 DX、MF、QA、OBS、EXC 等 Constitution Check，各门禁将在规划阶段落地到具体设计。 |
| DX，Developer Experience                | 开发者体验，指开发人员在使用工具、框架和流程时的整体感受和满意度，如 DX-01 是为开发者体验项门禁编码。                                       |
| MF，Manifests                           | 清单，元数据，例如版本信息、依赖关系和配置设置，如 MF-01 是为清单项门禁编码。                                                   |
| QA，Quality Assurance                   | 质量保证，确保软件产品符合质量标准和用户期望的过程，如 QA-01 是为质量保证项门禁编码。                                               |
| OBS，Observable                         | 可观测性，指系统能够提供足够的信息以监控和诊断其运行状态，如 OBS-01 是为可观测性项门禁编码。                                           |
| EXC，Exceptions                         | 异常，或例外，指系统在运行过程中可能遇到的例外、错误或异常情况，通常需要特别处理或记录，以确保项目的合规性和透明度，如 EXC-001 是为例外项门禁编码。               |
| Checklist                              | 清单，列出需求或任务的核对项，以确保完整性和一致性，如 CHK001 是为清单编码。                                                   |
| US，User Story                          | 用户故事，一种简短的描述，说明用户需求和期望的功能，如 US1 是为用户故事编码。                                                    |
| FR，Functional Requirements             | 功能需求，描述系统应具备的功能和行为，如 FR-001 是为功能编码。                                                          |
| NFR，Non-Functional Requirement         | 非功能需求，描述系统的性能、安全性、可用性等方面的要求。                                                                 |
| Entity                                 | 实体，指系统中的主要对象或概念，如用户、订单、产品等，如 Entity1 是为实体编码。                                                 |
| A，Assumptions                          | 假设，指在规格或规划中所做的前提假设，用于指导设计和实现决策，如 A-001 是为假设编码。                                               |
| SC，Success Criteria                    | 成功标准，定义需求或任务成功完成的条件。                                                                         |
| P，Priority                             | 优先级，表示需求或任务的重要性和紧急程度。                                                                        |
| Phase                                  | 阶段，指项目开发过程中的不同阶段，如需求分析、设计、实现、测试等，如 Phase 1 是为阶段编码。                                           |
| T，Tasks                                | 任务，指具体的工作项或行动步骤，用于实现规划和规格中的要求，如 T0001 是为任务编码。                                                |
| MVP，Minimum Viable Product             | 最小可行产品，指在开发过程中，为了验证需求或功能，而创建的最小可行的产品。                                                        |
| PRD， Product Requirements Document     | 产品需求文档。                                                                                      |

## 总结

为了在项目中尝试《Specification-Driven Development》方法论，按里面"九项原则"中的第一项原则 ------ "Article I: Library-First Principle"，找了一个具有清晰边界和最小依赖关系的业务场景，整个过程，从规格创建、澄清、规划、任务分解到最终实现，都是围绕规格进行，可行性较高。

简单场景，还是可以采用"Sequential Thinking" MCP，在对话过程结构化思考；或者尝试 AI 助手内置的 Plan Mode；但对于复杂场景，建议采用"Specification-Driven Development"方法论，以结构化、文档化的方式引导 AI 助手。
