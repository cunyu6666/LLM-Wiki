# 从设计到编码：设计不再困在 Figma 里。来自Cursor设计负责人 Ryo Lu

**如果你是设计师，大概经历过这样的日常：** 工程师好不容易把页面做出来了；你打开一看：

字重不对、按钮太挤、间距少了 4px；

动效不对味、节奏不顺畅；

然后开始截图、红线、写备注："这里偏了 50 像素，这里字号不对，这里不该有阴影......"

改完一轮再丢回去，工程师心里想的是：

"就这？要我改一晚上？"

这就是过去十几年里，设计师与工程师合作的基本剧本：

**设计师在 Figma 里画 → PM 写 PRD → 工程师实现 → 来回对线。**

**问题是------**

•Figma 里的稿再精细，也只是静态截图；

•PRD 再长，也承载不了"手感、节奏、氛围"这些东西；

•真正的体验，只会在代码跑起来、用户点下去的那一刻才显形。

而**在这一波 AI 工具------尤其是 AI IDE 出现之后，这条链路被悄悄改写了** ：

•设计师不再只能把"理想状态"画在图里；

•他们开始直接走进 IDE，用自然语言 + 一点点代码，**把想法投射到真实、可操作的界面里。**

在这样一套新范式里，

Cursor 的第一个设计师------Ryo Lu ，

在 Peter Yang 的访谈里，Ryo 一边 live demo 自己做的 ryOS（复古个人操作系统），一边讲 Cursor 的工作方式、AI 团队的组织形态，以及他眼中的"未来设计师"。


**一**

**Cursor 第一个设计师**

**Ryo Lu 是谁**


Ryo 现在是 Cursor 的 Head of Design，也是公司第一个设计师。

在 Cursor 之前，他在 Notion、Stripe 做产品与设计，对"工具如何让普通人做出复杂软件"这件事，一直有执念。

如果用一句话概括他现在在 Cursor 做的事：

过去 IDE 是工程师专属工具，

**现在他在尝试把 IDE 变成任何会表达「想法」的人都能使用的 AI 画布。**


**划重点**

**把 IDE 变成任何会表达「想法」的人都能使用的 AI 画布。**

**二**

**ryOS：一个个人 OS，如何成为 AI 能力的"实验场"**


Ryo 的 ryOS，看起来像一台被放大在浏览器里的老 Mac：

•像素风窗口、菜单栏、Dock；

•聊天 App、浏览器、iPod、虚拟 PC、扫雷、计算器......

•甚至可以在一个 DOSBox 里跑起"虚拟电脑"。

但它并不是一开始就这么完整。

最初，它只是 Ryo 为 Notion 同事写的一个小小 soundboard 工具------

放各种奇怪的声音，让大家"想起他"。

从这个极小的起点开始，它一步步长成了实验场级别的 OS：

**1.从默认 UI 到复古风**

•起初只是 Tailwind + shadcn 的默认样式；

•Ryo 一句 prompt："别做这种中性审美，试着做得更像经典 Mac OS。"

•Agent 给出了第一版复古样式：像素字体、浅色窗体，开始"有点味道"。

**2.从单一 App 到系统级体验**

•加了窗口、菜单栏、Dock；

•加聊天 App、浏览器、iPod App（完整复刻转盘操作，还能显示歌词和 MV）；

•再加虚拟 PC 和各种小游戏。

**3.从作品到实验室**

•ryOS 成为 Ryo 每次测试新模型、新功能（Plan Mode、多 Agent）的"试验田"；

•他甚至做"UI 考古"：买老 Mac 机器回家研究；上老 UI 资料网站看像素级截图和图标；用这些历史细节，对齐自己的复古风。

对我们这些普通人来说，**ryOS 其实给了一个很重要的启示：**

AI + 代码学习，不一定要从"造轮子"开始，

可以从一个小而真、有趣又可叠代的个人系统开始。


**划重点**

**AI + 代码学习，不一定要从"造轮子"开始，** **可以从一个小而真、有趣又可叠代的个人系统开始。**

**三**

**Cursor**

**角色高度模糊的 AI-native 组织长什么样？**


听 Ryo 描述 Cursor 内部，会发现一个明显不一样的气质：**角色边界被故意做得非常模糊。**

**1. 他们几乎没有传统意义上的 PM**

•没有那种专职写 PRD、画路线图、开需求会的 Full-time PM；

•有一位工程师出身的前创始人，后来慢慢接了一些 PM 职能；

•但大部分"产品管理"工作，被拆散嵌入到每个 Builder 身上。

**"很多 PM 的工作，其实是分散在整个 Builder 团队里的。"**

**2. 每个人都是 Builder，而不是只守着一个「工位」**

•有人更擅长视觉和交互；

•有人更擅长架构和高可用；

•但**大家共同的日常都是：看到问题 → 想方案 → 在 Cursor 里拉起 Plan → 带 Agent 写代码 → 自己审。**

角色不是"UX、PM、FE"那种刚性标签，而更像：

**一群能 end-to-end 解决问题、顺便会用 AI 带团队的 Builder。**

**3. 他们不迷信「一年 Roadmap」**

在模型每天都在变化、新框架不断涌现的环境里，

太硬的路线图往往是负担而不是资产。

**Cursor 更重视的是：**

**•一个模糊但清晰的方向：** "让任何人都能构建任何东西"；

**•当前状态和目标之间的一些"台阶"** ；

•**确保：** **每一步都是朝大方向走的真迭代** ，而不是纸面规划。

这对国内很多团队、尤其是「设计 / 产品 / 工程三区分明」的公司，其实是挺尖锐的一记提示：

在 AI-native 团队里，你可以保留头衔，

但你必须接受：**真正的价值来自你亲自带着 AI 把事做完，**

**而不是认领一块"职能圈地"。**


**划重点**

**真正的价值来自你亲自带着 AI 把事做完，** **而不是认领一块"职能圈地"。**


**四**

**Plan Mode：「从 vibe 到 Spec 再到 Code」的三段跳**


Ryo 在节目里演示了一个非常具有标志性的场景：

在 ryOS 里，**现场用 Plan Mode + Agent** ，加一个新的"复古计算器" App。

整个过程可以拆成三段跳：

**1）先说 vibe，而不是 UI 细节**

Ryo 没有从"按钮样式、字号、栅格"开始，而是先讲：

•这个 App 的气质：日常、耐看、不过时；

•要融入 ryOS 的整体复古风格；

•不要那种"看一眼就知道是 2025 年 AI demo"的炫技感。

也就是说：

**先用人话讲清楚"感觉"和"边界"，不急着讲组件。**

**2）让 Plan Mode 把 vibe 翻译成 "可审阅的 Spec"**

Plan Mode 会先做几件事：

•基于当前代码库、主题、组件，提出多个 App 候选（计算器、纸牌、时钟......）；

•选中"计算器"后，产出一份细致计划：

•功能：支持哪些运算、有哪些错误和边界；

•UI：窗口布局、按键区、显示区；

•技术：改哪些文件，创建哪些组件，状态放在哪。

你看到的，不再是一坨聊天记录，而是一份结构化的 Spec 文档。

**3）在人类熟悉的"文档层"审稿，然后再让 Agent 写代码**

Ryo 做的不是 dive 进代码，而是：

•在 Plan 文本里直接改：删功能、改措辞、补充约束；

•比如："先用占位图标，之后去 icons 文件夹替换"；

•Plan 看顺眼了，再点 Build，让 Agent 按方案改代码。

出错时，他也不是直接 debug，而是丢一句：

"请根据浏览器里的报错和行为，帮我修好这个问题，并告诉我你做了什么。"

Agent 会去读 log、读错误信息，再自行修复。

**这套模式的关键在于：**

**人类退到一个更高的抽象层------**

**不再写满细节，而是专注于「方向对不对」「约束完不完整」，**

**再把"执行"交给 Agent。**


**划重点**

**人类退到一个更高的抽象层**

**不再写满细节，而是专注于「方向对不对」「约束完不完整」，**

**再把"执行"交给 Agent。**


**五**

**多 Agent 协作：把 AI 当成「虚拟小团队」来带**


很多人现在用 AI 的方式还是：

"一个大模型干所有活"------想点子、写方案、写代码、修 bug、润 UI，**全靠一个 Chat 硬撑** 。

结果经常是：

输出风格混乱、代码难以维护、每次问它"你刚刚干了什么"也说不清。

**Ryo 的思路更像是：**

**把多个 Agent 视作一个「虚拟小团队」，**

**每个 Agent 有明确分工，人类负责 orchestrate编排。**

举个简化版的例子：

•Agent A：负责 Plan / 规格文档；

•Agent B：负责具体实现（组件、逻辑）；

•Agent C：负责视觉微调、对齐 design system；

•Agent D：负责 refactor / 清理代码。

**他的日常工作流是：**

1.在本地用一个主 Agent 驾驶全局；

2.大量"小而清晰"的任务丢给后台 Agent 或云端执行；

3.最后在 PR 或 Plan 层做统一审阅和合并。

对我们来说，这一段带来的认知升级是：

**"会写 prompt"只是入门，**

**真正需要练的，是如何拆任务、设接口、定义验收标准，让一群 Agent 配合你工作。**

这其实已经非常接近「管理一个跨职能小团队」的思维方式了。


**划重点**

**"会写 prompt"只是入门，**

**真正需要练的，是如何拆任务、设接口、定义验收标准，让一群 Agent 配合你工作。**


**六**

**避免"紫色 AI 土味设计"：关键不在 prompt，而在设计系统**


你肯定见过这种 UI：

•大号紫色渐变背景；

•毛玻璃 + 霓虹边框；

•每个角都在发光，看上去"很 AI"。

Ryo 把这种东西叫做 "AI design slop"：

并不是不能炫，而是那种"一眼就知道是模型默认模板"的味道。

他的观点是：

**真正决定审美下限的，**

不是你那句"帮我做一个酷炫现代 dashboard"的 prompt，

**而是你有没有先给 AI 准备一套严肃的 design system。**

**他的实践路径：**

**1.用成熟组件库打底**

•大量 UI 基于 shadcn + Radix：

•按钮、菜单、Tabs、对话框、Popover......

•这些组件在 accessibility、键盘导航、状态管理方面已经很成熟。

**2.在此之上构建主题系统**

•自己再加一层 theme：颜色、圆角、字体、阴影、间距；

•用这些主题来定义 ryOS 的复古气质、不同皮肤。

**3.让 AI 在"规则"内创作，而不是在真空里乱飞**

•Prompt 不再是"随便做个好看 UI"，

•而是"在现有 ryOS 主题和组件体系里，增加一个百分百符合风格的新窗口/控件"。

换句话说：

**AI 更像一个「懂规矩的初级设计师」，**

**真正负责美学底线和统一性的是那套你搭好的设计系统。**


**划重点**

**AI 更像一个「懂规矩的初级设计师」，**

**真正负责美学底线和统一性的是那套你搭好的设计系统。**


**七**

**给设计师 \& PM 的具体行动路径**


讲了这么多，如果落回到"我明天要干点什么"，Ryo 的建议其实朴素到有点可爱：

**"下载 Cursor，找一个小东西做出来。不要害怕。"**

我们把它拆成一套可以直接照做的三步路：

**第一步：选一个「足够小，但真的有用的」项目**

不要从"系统重构""公司中台"这种庞然大物开始。

而是：

•个人主页 / Dashboard；

•小小的习惯打卡工具；

•一个属于你的迷你 OS（哪怕只有 2--3 个 App）。

标准只有两个：

1.真能跑；

2.你自己会用。

**第二步：为 AI 准备一套「你的设计系统」**

不需要一开始就造出 Notion 那种系统，但至少做到：

•选一个基础组件库（shadcn / Ant / Chakra / Tailwind + 自建组件）；

•定下：颜色体系；字体层级；圆角、阴影、间距；

•用文档或代码注释写给 AI 看：有哪些组件；哪些变体；哪些 tokens 能改，哪些不能动。

这一步，会让你的作品从"AI 泥浆风"一夜之间变成"起码看着顺眼"。

**第三步：刻意练习「vibe → Plan → Code」完整闭环**

找一个你已经有点感觉的需求，完整走一遍：

1.写 vibe：谁在什么场景用、希望有什么感觉、绝对不想要什么；

2.用 Plan Mode（或类似能力）生成 Spec / 计划文档；

3.在文档层删改、补充，直到你愿意"为它签字"；

4.交给 Agent 实现，然后你做最后一轮体验 + 代码层审阅。

**如果你每月能完整走通 1--2 次这样的闭环，你的职业身份会慢慢发生位移：**

**•从"只会画 Figma、写需求"的那个人，**

**•变成那个把模糊机会变成真实产品的人。**


**划重点**

**下载 Cursor，找一个小东西做出来。**

**不要害怕。**


**结束语**


上一代互联网告诉我们的是：

"人人都要懂一点产品思维。"

这波 AI 带来的新要求，可能变成：

**"人人都要懂一点 AI + 代码 + 系统思维。"**

你的职位名称可以暂时不变------

设计师也好，PM 也好，工程师也好。

但你真正的职业安全感，会来自这一点：

你敢不敢，

也愿不愿意，

**亲自站到「vibe → Plan → Code」这条链路的最前面，** **然后带着一群 Agent，把事做完。**
