# OpenAI 设计师：不写代码的设计师，将成为团队瓶颈

OpenAI Codex 团队的设计主管 Ed Bayes，现在 70-80% 的时间在写代码。

他没转行，是设计这个职业本身变了。

![](https://cubox.pro/c/filters:no_upscale()?imageUrl=https%3A%2F%2Fmmbiz.qpic.cn%2Fsz_mmbiz_png%2FZKqVLiaIpzFnj8PwVcSdPweu3LE8vo5fVBrXT762AOFU5qDENFqYG5evFlNFRgTrYs6iaEZM7XdVuicqdjzYEsrC5wIuIOibhDuJ08yFl2ia4OgU%2F640%3Fwx_fmt%3Dpng%26from%3Dappmsg%26watermark%3D1%23imgIndex%3D0)

最近，产品增长博主 Aakash Gupta 请来了 OpenAI 的 Ed Bayes 和 Figma AI 设计总监 Gui Seiz，做了一期将近一小时的深度对谈。

这一期聊的核心问题是：

**当 AI 编程工具和设计工具之间的墙被打通之后，设计师的工作流程到底变成了什么样？**

这期播客信息密度相当高，从实操 demo 聊到了组织变革。我把关键内容整理了出来。
01

## 别选了

在 AI 设计能力越来越强的当下，设计行业开始出现了一个争论：到底应该用代码主导设计（比如直接在 Codex 里写），还是应该坚持传统的画布流程（在 Figma 里画）？

Gui 的回答是：
> " 我们终于到了一个不用选的时代。画布依然是快速发散、横向探索的最佳工具。而代码让我能够迅速把想法变成真实可交互的东西。以前我必须二选一，现在不用了。

Ed 补充了一句：
> " 这其实是个伪命题。你是创作者，你有一整套工具箱，关键是为手头的任务选对工具。

他俩的核心观点是：**画布和代码不再是两条路，而是同一条路上的两种交通工具，随时切换。**
![](https://cubox.pro/c/filters:no_upscale()?imageUrl=https%3A%2F%2Fmmbiz.qpic.cn%2Fmmbiz_png%2FZKqVLiaIpzFk3X6CRFGpvpiahQs1OBGZDWxicGq5d7eUqpteEqnicbfIaadS3kpaXoum3WiajkoibdGJbVcUUiaw7ibjFjQo3QsiarD6AhzJhdZ0lf50%2F640%3Ffrom%3Dappmsg%26watermark%3D1%23imgIndex%3D1)岔路合一

那......具体什么时候用哪个呢？

Gui 给了个实用的判断框架：如果你在打磨一个组件的细节，可以直接从代码开始。如果你在规划一整个页面流程，画布可能更直观。如果你想打破某个交互范式、做大量发散探索，那就用更低保真的方式。

**重点是：你的低保真起点，不再必须是马克笔画的草图了。**

它可以是一个在 Codex 里几分钟搭出来的可交互原型。
02

## 一键同步

说到这，不少人的第一反应应该是：代码和设计稿之间来回切换，精度损失怎么办？

Ed 直接做了个 demo。

他打开 Codex 桌面端，加载了一个叫 Codex Composer System 的项目。这是他之前在 Figma 里设计好的一个 composer 组件，但他想在代码里测试一些交互细节，比如按钮之间的变形动画、不同状态下的尺寸变化、tap target 的位置。

测试完之后，他想回到 Figma 做像素级的精修。

操作很简单：在 Codex 里对它说「把我的首页和几个 composer 状态导入到一个新的 Figma 文件里」，然后 tag 上 Figma。

Codex 先读取 React 应用的文件结构，然后调用 Figma MCP，自动截取页面快照，生成一个 Figma 文件。再点击「在 Figma 中打开」，所有内容就到了 Figma 里。

**而且是像素级精确的。**

padding、border-radius、阴影的 x/y 值，全部和代码里一模一样。

以前工程师得一个个去量、去对数值，现在，开箱即用。
![](https://cubox.pro/c/filters:no_upscale()?imageUrl=https%3A%2F%2Fmmbiz.qpic.cn%2Fmmbiz_png%2FZKqVLiaIpzFmbrywPlPlrSaMgAU6y0SKicTUWH7VcEqF3ibEQHrvGMUOV5cJ2UR1uMX6XVe4XNictkvWOEcO5hDg31jWAeqHrTWFcqEhxkGCZvY%2F640%3Ffrom%3Dappmsg%26watermark%3D1%23imgIndex%3D2)双向传送门

更酷的是反向操作。

在 Figma 里改了颜色、调了间距之后，复制这个组件的链接，粘贴到 Codex 里说「用这个更新我的代码」，代码就自动同步了。

Gui 还提到一个细节：你不需要导入整个页面，可以选择具体的节点。比如只要那个文本输入框，不需要后面的大白色背景。

**设计师不会写代码？没关系，这个流程不会卡住你。**   

你在 Figma 里改，工程师在代码里收到更新，协作不再是两个孤岛。
03

## 越来越准

那......实际用起来，会不会翻车呢？

Gui 坦言，确实有些边界情况：shader 效果、复杂的 CSS transition，静态画布暂时还搞不定。

不过他紧接着说：
> " 从这里开始只会越来越好。AI 模型每进步一次，这一切就好一分。我们只是搭好了管道，精度会自动跟着模型一起涨。

而且 Figma 最新的工具做了个关键升级：**导入的组件不再只是网页截图了，而是能引用你的 design token 和本地样式库。**

也就是说，你拿到的并非一个「长得像」的副本，而是真正使用了你的设计系统的组件。

Ed 还给了个实用的建议：把 agent 当同事来看。

如果你的 Figma 文件命名混乱、组件没整理好，同事上手也费劲，AI 也一样。**把组件命名规范、design token 和 CSS token 对齐，这些基础工作做好了，不管是人还是 agent 都能顺畅协作。**
![](https://cubox.pro/c/filters:no_upscale()?imageUrl=https%3A%2F%2Fmmbiz.qpic.cn%2Fmmbiz_png%2FZKqVLiaIpzFntNKausavrYIHUFViaS9mTwmLN3ZW9lkp31R3d4RRtRrscWeHLvcbWm0dBFca2L5EUia4oYybVUSkdCI8OLzhZJ0NzK2gHD6w3I%2F640%3Ffrom%3Dappmsg%26watermark%3D1%23imgIndex%3D3)

他还提了一嘴：OpenAI 最新的 5.4 模型在配合 Figma MCP 时，表现比之前好了一大截。好几个内部设计师主动跑来跟他说，这个模型用起来感觉完全不一样了。
04

## 都在写代码

那在 OpenAI 和 Figma 内部，实际的工作方式到底变成什么样了呢？

Ed 说，Codex 团队本来就偏技术，设计师大多会写代码。但变化也在整个公司蔓延。
> " 在我们的工作进展 Slack 频道和设计评审里，我看到越来越多人带着一种混合产出来。会有一个 Figma 链接，但旁边还有一个 URL，大家点进去就能看到可交互的版本。

**甚至连 UX 文案设计师都开始自己提交 PR、把改动推到生产环境了。**
![](https://cubox.pro/c/filters:no_upscale()?imageUrl=https%3A%2F%2Fmmbiz.qpic.cn%2Fsz_mmbiz_png%2FZKqVLiaIpzFlbqTcibQQEtJMY8vJKS0uhYa8H6EYhV27SX3phYP8dmrIVj4GMiaTKg8GKm2nRB2t7yOyIyWTHCmmicB0d16THxhlv8acWYLOibkU%2F640%3Ffrom%3Dappmsg%26watermark%3D1%23imgIndex%3D4)文案设计师提 PR

他回忆说，转折点大概在去年 12 月。

当时他们在内部试用 Codex 桌面端，模型能力突破了一个临界点：
> " 突然感觉，任何人都能跳进来改东西了。

Figma 这边，变化则可能更戏剧性。Gui 说，去年年中他还在用传统流程：设计 → 原型（用 Figma Make）→ 交给工程师实现。

现在呢......
> " 我们可能先在生产环境里加一个 feature flag，然后在 Make 里打磨一些细节，再用 Codex 把改动写回 GitHub。工具之间不停地跳来跳去，哪个顺手用哪个。

最让他意外的是这种工作方式在 Figma 内部的传播速度。一开始只是偏技术的工具团队在用，现在连做商业化变现的团队里，设计师都在搭技术含量相当高的原型。
> " 以前，设计师可能不会允许自己去做这些事。现在他们不光有了词汇量，还有了实际的能力，可以不依赖工程团队就把东西做出来放到人面前。

Gui 半开玩笑：
> " 我们内部有个梗，P2 需求不存在了。因为现在什么都可以直接 ship。
05

## 设计成瓶颈？

不过 Ed 也指出了新的问题。
> " 如果开发者的效率被 AI 提升了 10 倍，我觉得这不算夸张。那设计师呢？可能被提升了 1.5 到 2 倍。
![](https://cubox.pro/c/filters:no_upscale()?imageUrl=https%3A%2F%2Fmmbiz.qpic.cn%2Fmmbiz_png%2FZKqVLiaIpzFnjPLLzRRj4cY0t9LDvJv6hTicnb7o9khY2ickcQ8BKKBafZxcJ4OL7JA0KZjOSTz0pxIC1vLdpdXjsHPuibmIEvbpicVjkszztAq4%2F640%3Ffrom%3Dappmsg%26watermark%3D1%23imgIndex%3D5)速度落差

**这意味着，如果设计师自己不写代码，设计反而会成为整个团队的瓶颈。**

这个判断，还是挺真实的。

以前大家觉得工程是瓶颈，设计出完稿等工程实现。现在反过来了，工程跑得飞快，设计跟不上。

Ed 说，挑战在于：你得知道什么时候该追上工程师的速度，什么时候该刻意慢下来，走一个深度设计流程。

Gui 也有类似的感受：
> " 你几乎是被赋能到了一个程度，以至于你得开始想「我应该做什么」，而不只是「我能做什么」。
06

## 五步上路

那对于还在传统公司、合规行业（医疗、金融）的人来说......怎么开始呢？

Ed 的建议非常朴素：
> " 就去试试。这话听着简单，但真的就是这样。

他举了几个身边的例子。

OpenAI 市场团队的一个同事，不懂 iOS 开发，自己用 Codex 做了个 iOS app，装在手机上拿到公司活动上展示。品牌传播团队的同事想设计一个活动座位表，就让 Codex 生成了一个可以拖拽姓名的 HTML 文件。

Ed 自己马上要去日本度蜜月，他和妻子正在一起用 Codex 做一个小应用，标注想去的咖啡馆和餐厅。他妻子完全不是技术背景，但也在上手操作。

Gui 的建议则是：
> " 你甚至不需要从代码开始。我们所有人手机上都有 ChatGPT。你能提问，你就能做事。

他回忆自己以前学编程的痛苦经历，报各种 bootcamp，纠结该把有限的时间投入到哪个方向。
> " 现在不用了。我可以直接问：能帮我搭这个吗？如果它开始做了，我再问：这是怎么工作的？我想深入多少就深入多少。如果我不关心实现细节只想要结果，也完全可以。
![](https://cubox.pro/c/filters:no_upscale()?imageUrl=https%3A%2F%2Fmmbiz.qpic.cn%2Fmmbiz_png%2FZKqVLiaIpzFkYCIfwESeFrAbUtmJm9eerwbF7ucIAhu2JicptnxiaSDnDulBvClqJPOtsslf6YoicktCP5IPJFaD4PNrVgqibnPLNwWa1964Alms%2F640%3Ffrom%3Dappmsg%26watermark%3D1%23imgIndex%3D6)五步上路

综合两人的建议，大致可以归纳为五步：

• 下载工具，随便搭个东西，哪怕是给自己用的小玩意

• 跑通一次完整循环：导入一个页面，编辑它，再推回代码

• 从打磨细节开始：先用代码处理动画微调、加载状态这些

• 改变起点：从「在 Figma 画线框图」变成「在 Codex 搭可交互原型」

• 把 AI 当导师：不懂就问，逐步建立对系统的理解
07

## 全能足球

聊到这里，一个避不开的问题终于浮出水面：设计师、工程师、产品经理，这三个角色的边界到底还存不存在？

Gui 用了一个足球类比：
> " 1970 年代荷兰有一种踢法叫「全能足球」，每个球员都能踢任何位置。现在我们就像那样，当一个人移到那边去了，你能补过来。作为一个团队，你的威胁大了很多。
![](https://cubox.pro/c/filters:no_upscale()?imageUrl=https%3A%2F%2Fmmbiz.qpic.cn%2Fmmbiz_png%2FZKqVLiaIpzFmUkziaIQMnIRFsPKUvCZ16fY40hb4ZEnuNZzYfwta8yVLytNaVvmA3EBslCNgl7fib3kSkTNoVgCs0uiaEJCEgaLbviaDcaUma6DY%2F640%3Ffrom%3Dappmsg%26watermark%3D1%23imgIndex%3D7)全能足球

他在 Figma 看到的现实是：设计师在提交代码，产品经理在做原型，而且大家还在写 AI Skills 来指导模型如何做设计决策、如何做产品判断。
> " 设计师可以借助 PM 的思维模式来做决策，PM 在试图搞清楚怎么调试代码。每个人都在借助别人的「尖峰能力」。

不过 Ed 有不同看法：
> " 设计师的角色不会消失。产品经理不会消失。工程师不会消失。因为这些角色在组织里承担的使命是不同的。

他的逻辑是：设计师是用户的代言人，工程师关注系统和性能，产品经理的使命是构建伟大的产品。工具箱可能一样了，但组织对你的期待没变。

**用 Ed 的话说：即使我现在大部分时间在写代码，我依然认为自己是一名设计师。**

Gui 则从另一个角度解释了为什么角色不会合并：
> " 你天然地被某些事情吸引，这塑造了你的职业路径。以前，学一个新工具的时间成本太高，你被困在自己的领域里。现在抽象层降低了，你能踏进别人的领域。但这不意味着你会变成 PM 或工程师，而是你不再只看到三分之一的画面了。

Ed 最后提了一个在 OpenAI 内部已经在发生的变化：
> " 我们团队有个说法，prototypes not PRDs。产品经理不再写长长的需求文档，而是带着一个可运行的原型来讨论。他们甚至会提交一个 PR 来压力测试自己的想法，然后再交给工程师接手推进。
08

## 好奇心定胜负

最后，Gui 总结称：
> " 好奇心，会是决定谁能在这个新时代成功的核心能力。AI 变化太快了，你只需要有足够的好奇心去问一句：这对我正在做的事情意味着什么？

他说，现在每个人身边都有一个「极其耐心的导师」，永远不会累，永远不会下班，你可以随时让它用更简单的话再解释一遍。
> " 如果你能培养出这种好奇心的本能，那就是「如何开始」的答案。

**而如果你觉得，这只是一个可以小幅适应的变化，**

**那你可能会，错过真正重要的东西。**
