# Dan Carey：Claude Design 从想法到上线

## 一、Anthropic Labs：押注工厂

Anthropic Labs 是"前沿实验室里的实验室"——我们叫它**bet factory（押注工厂）**。

**做法**：
- 同时有十几个极小的团队
- 在探索模型能力的前沿
- 下注某个东西能不能成
- 管用的加倍下注，不灵的收摊

**出过的产品**：Claude Code、Claude Design、MCP、Skills、Cloud、Chrome 扩展、免提音频——**全是从 Labs 出来的**。

**最大特点**：我们**跑 loop 的速度**。

具体方法：

| 维度 | 做法 |
|------|------|
| 用户反馈 | 每天跟用户在一起，最喜欢听"请来吐槽我" |
| 研究员对齐 | 每天跟 Anthropic 研究员在一起，最喜欢问"最近有什么让你惊讶的" |
| 发版节奏 | 每隔一两天就发新东西；用户当天反馈，当天就能回应 |
| 预测未来 | **不预测未来**——不断发布、看、学、再发布 |
| Loop 频率 | Claude Design 项目 10 周内把 loop 跑了 50 到 100 次 |

## 二、为什么做 Claude Design：瓶颈在转移

**最直接的原因**：Claude Code 把工程师变得非常非常快，**我们其他人得跟上**。

### 产品开发周期压缩史

```
以前是 6 个月。
后来是 1 个月。
再后来是 1 周。
现在是 1 天。
```

> 以前要花一周的事，现在几小时搞定，直接推到用户面前。

### 瓶颈转移

> Claude Code 起飞后，瓶颈就移了。
> **瓶颈从"做功能"移到了"想清楚该做什么"。**
> 在很多场景下，后者才是真正难的事。

要么跳过早期步骤、边走边定，可能做出错的东西——要么**想办法让其他角色也快起来**。

> 我们的设计师、PM 都快跟不上了。
> **我们需要自己的加速工具。**

### Nate 的故事

团队里有个叫 Nate 的人，之前是 Claude Code 的设计师。他**亲眼看到这些团队被加速到这种程度时会发生什么**。

他花了一个周末边做另一个押注边拼了个原型。

**非常简单**：
- agent SDK
- 一个极薄的 IDE 壳
- 调用的 skill 是他之前在 Claude Code 里用过的那个

> 我们很多 labs 押注就是这么起步的。
> **一个人，一个周末，一个屏幕，边做边报。**

他把这个原型丢到 Slack 里，大家热心地补意见——哪些是亮点，哪些是"完全坏的快去修"。**这就是他前几周的路标**。

## 三、流程：原型 > PRD > 视频/分支

**Claude Design 项目里没做的事**：
- 没提前写 PRD
- 零愿景文档
- 零 KR 会议
- 没年度人头规划
- 没"未来两年怎么做"的两页纸
- 没提前把要发的稿子写好

> 如果你已经明确知道要做什么，这些流程挺好用。**但我们当时并不知道**。我们只知道——有那么个火花。

### 原型 > 文档

> 我们喜欢用原型，因为文档不精确。
> 两个人看同一份文档，脑子里想的可能是两个不同的产品。
> 而且通常这两个想法都不是作者写文档时脑子里的那个。

原型**更具体、更直观、能亲手把玩、真正感受体验**。

### 原型迭代周期：几分钟

> 在这个项目里，我们把原型迭代周期压到了**几分钟**。
> 谁有个想法，几分钟就能做出原型推出去。

最简单的做法：
1. 和团队其他人聊
2. 录下来
3. 转成文字
4. 主要聊：问题是什么 / 解法大致做什么 / 为什么你关心这个问题
5. 转录稿喂给 Claude Design
6. 让它给几个方案

> 这套对一个写了快 20 年 PRD 的 PM 来说，**基本替代了 PRD**。

### Pitch-off 仪式

在 Labs 里我们时不时会做这个仪式——聚在一起，叫 **pitch-off**。

> 我喜欢这个仪式，因为大家聚在一起，头脑风暴。你基本就是想方设法吸引别人来跟你一起做你那个——你觉得团队该做的事。

第一次在 Claude Design 项目上做 pitch-off 的时候，**后半场 100% 的 pitch 都是用 Claude Design 现场做的原型或幻灯**。

> 都是会上现场做出来的。
> 那一刻我们就确信——**这个押注值得加倍**。

## 四、小团队的力量：3 人 + Claude，角色界限消失

### 团队规模

> 几乎所有 labs 押注都从**一个人**开始。
> 就是一个人，带着好搭档 Claude，去探一个特别难的问题、特别难的点子。

这个阶段，你不是要做世界上最好的产品。你只是在**找那一下魔法时刻**、**找那一丝热度**。

> 你在找：嘿，这个想法有戏，有点小火花——我们能把它做成一个好产品。

**坦白讲，大部分 labs 押注走不到这一步**。大部分我们会在这之前就收摊。早期探索大部分就是几小时或几天。

### 团队扩展曲线

```
1 人（找火花）
  ↓
3 人（300% 扩展）
  ↓
5 人（发布前）
```

> 我们通常把团队大幅扩展——300%——到 3 个人。**所以都不是大团队。**

这样做的原因：

> 我们希望一个紧密的小组一起探索，**协作开销极低**。

### 3 人 + Claude

> Claude Design 大部分开发阶段，只有 3 个人 + Claude。
> 万幸 Claude 是个不错的队员。

**什么让这成为可能**？在座很多 solo builder 肯定也深有体会：

> 团队里每个人什么都干。
> **工程师跟用户聊，PM 写代码，设计师做数据分析**。
> Claude 让这些成为可能。

### 角色界限消失

> **角色之间的界限，在这个团队里，已经基本消失了。**
> 你当然还有自己的专长、有独特的视角、有你的多样性。
> 但任何时刻，这个团队里任何一个人，都可以去跟十个用户聊，搞清底层问题，设计一个解法，推到用户，听反馈，继续迭代——需要的话，完全 solo。

> 团队里大部分东西都是纯 solo。
> 每个人都能干所有事，这是我们降低协调开销的关键。

## 五、可直接塞进演讲的核心金句

### 最高级金句（可直接砸）

> "**Claude Code 起飞后，瓶颈就移了。瓶颈从'做功能'移到了'想清楚该做什么'。**"

> "**我们的设计师、PM 都快跟不上了。我们需要自己的加速工具。**"

> "**角色之间的界限，在这个团队里，已经基本消失了。**"

### 数据锚

| 数据 | 用法 |
|------|------|
| **3 人 10 周 出一个产品** | 对位你的 QoderWork 5 人 7 天 |
| **50-100 次 loop 跑通产品** | 对位你的 monorepo 71 次 commit |
| **3 人 + Claude = 全角色覆盖** | 对位你的 "1 人覆盖 3 角色" |
| **产品周期 6 月 → 1 月 → 1 周 → 1 天** | 对位"Designer+Builder 的时代窗口" |
| **原型迭代周期压到几分钟** | 对位你的 Design Desk 10 秒 |

### 故事锚

**Nate 的故事** —— Anthropic 的 Nate 之前是 Claude Code 设计师，**一个周末用 agent SDK + 极薄 IDE 壳 + Claude Code skill 拼出 Claude Design 原型**。

这把"一个设计师转身"的故事**钉到了行业顶级机构**的案例上。
