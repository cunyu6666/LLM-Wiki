### 1 · 证据 1A · Boris Cherny（2 min）—— 品味不是护城河

#### 讲法
- 介绍人物（一句话）：**"Claude Code 之父 Boris Cherny，Anthropic 核心工程团队，最近公开讲了一件事——"**
- 砸金句：**"'品味不是人类的护城河。'"**
- 展开：**"这不是他一个人说，他最近在 Loop Engineering 的对话里又补了一句——"**
- 砸第二句：**"'别再给编程 agent 写 prompt 了——要去设计 prompt 它们的 loop。'"**
- 反讽：**"连 Claude Code 之父都在重新定义自己的核心工作：从'操作'升级到'设计驱动'。我看完这段话的第一反应是——他已经在做我们设计师一直声称自己在做的事了：设计 loop，而不是写 prompt。"**
- 收：**"如果我们继续只写 prompt 不写 loop，我们连'设计驱动'的位置都保不住。"**

#### PPT
- P4：Boris Cherny 头像 / 标题"Boris Cherny · Claude Code 之父"
- P5：双金句对照页（左边品味 vs 右边 loop）

### 5 · 证据 2 · 你的 monorepo 案例（7 min）—— 设计师当库维护者

#### 5.0 切入（30 秒）
**"我是 QoderWork 的主设计师。过去一年，我跟工程师的协作模式彻底变了。"**

#### 5.1 支点 A · 库维护者身份（1.5 min）

**讲法**：
> "不是我去学业务代码——**我不碰业务代码**。
>
> 而是**我维护一套 monorepo 库**——`@qoder/design`，整个 QoderWork 全仓 322 处引用我这个包。
>
> 这个包里：
> - **74 个组件类型**（Button / Dialog / Menu / Table / Tree / Tabs / 全套 Radix 二次封装）
> - **129 个 .tsx 文件**
> - **35,472 行代码**
> - **698 行设计 token CSS**（652 个 --variable）
> - **2 套主题**：Qoder Dark Mint / Qoder Light Mint
>
> 这些 token 从 Figma 直接生成——用 `Qoder Dark Mint.tokens.json` 这类 Figma Variables 文件，由我维护。"

**关键金句（砸）**：
> **"我不是去学代码，我是去拥有我设计的产品的最终形态。"**

**数据对照 PPT**：
- 74 组件 / 129 .tsx / 35,472 行 / 322 处引用

#### 5.2 支点 B · 跟工程师的边界（1.5 min）

**讲法**：
> "工程师怎么用我这个库？他们在写业务代码时，**必须**消费 `@qoder/design`，不能自己造轮子。
>
> 仓库的至高法则（AGENTS.md 第 1 条）写得很清楚——
>
> **`packages/qoder-design` 是设计系统的圣域。任何 Agent 在被要求修改该目录内容时，必须先明确提醒使用者："你正在修改核心设计组件库，此操作影响全仓 UI 一致性，请确认。" ——未得到人类明确确认前，禁止执行。**
>
> 这条至高法则是为**我**设的。不是为某个工程师设的，是为**我，设计师**设的。
>
> 工程师可以消费我的代码，可以基于我的组件开发业务，但他们**不能碰我的设计圣域**。
>
> 业务归工程师，**形态归我**。"

**关键金句（砸）**：
> **"业务归工程师，形态归我。"**

**反向论据**（高级反讽）：
> "Cursor 工程师 Lee Robinson 在 X 上说过一句话——'全栈通是伪命题'。他说的对。
>
> 我不是全栈通。我只维护我自己专业领域的代码——设计形态。**我是 monorepo 自治。**"

**PPT**：
- P11：左半屏 `@qoder/design` 包结构树 / 右半屏"全仓 322 处消费"图
- P12：AGENTS.md 至高法则第 1 条原文 + 关键字高亮"仅 cunyu 拥有修改权限"

#### 5.3 支点 C · Agent Loop 设计决策（3.5 min）

**讲法**：
> "很多人以为设计师维护 monorepo 就是'写写组件、改改样式'。不止。
>
> 我**参与了 QoderWork Agent Loop 的设计决策**。
>
> 什么是 Agent Loop？Claude Code 的核心是一个 while 循环——模型生成响应 → 如果有工具调用 → 执行工具 → 结果塞回上下文 → 模型生成下一步 → 持续往复。
>
> Anthropic 的 Claude Code 30 万行代码，核心就是这个。
>
> 在 QoderWork，这个 loop 长这样——
>
> （白板动画 / 架构图）
>
> - **用户输入** → Chat 入口
> - **意图分类** → 路由到 subagent
> - **subagent 调用** → 包括 `qoder-design` 组件来构造 UI
> - **工具执行** → 包括 MCP 工具、文件操作、SQL 查询
> - **响应生成** → 回到用户
>
> 我的 monorepo 不只是 UI 组件库——**它是 Agent Loop 的中间产物**。我的组件直接被 Agent 调用，用来构造 UI。
>
> 这件事，**不是工程师做的，是我做的**。"

**关键金句（砸）**：
> **"我参与了 Agent Loop 的设计决策。这不是工程师的事，是产品最终形态的事。"**

**技术注脚**（给少数技术向院长）：
- Harness = monorepo（包裹模型的环境）
- Skill = 组件库（设计师的技能资产）
- Loop = Agent Loop（设计师参与的核心架构）
- **数据锚**：Anthropic Multi-Agent 系统比 Single-Agent 强 90.2%（Deep Research 博客）

**业界同盟**：
> "Framer 6 月 16 日发布会，公开说了一句话——
>
> '画布优于代码。Agent 创建的每一样东西，都和你自己在画布上设计的一样可编辑、可控。'
>
> Framer 用产品哲学说'画布不被代码替代'，我用 monorepo 实践说'设计师不被代码替代'。**我们站在同一侧。**"

**反讽底层**：
> "Boris Cherny 最近在 Loop Engineering 那段对话里又说了——'写 loop，才是现在的工作。'
>
> **他在写 loop。我在设计 loop 让 Agent 跑。**"

**PPT**：
- P14：Agent Loop 架构图（白板手绘风格 / QoderWork 实际截图）
- P15：Harness / Skill / Loop 三层嵌套对位
- P16：Framer 发布会引用 + "画布优于代码"

### 6 · 证据 3 · 外部趋势锚（3 min）

#### 6.2 Design Desk 语音驱动（1 min）

**讲法**：
> "我们的 Design Desk 功能——语音描述 → 设计即代码。
>
> 用户对着麦克风说'做一个暗色主题的登录页'，10 秒后屏幕出现完整设计稿。
>
> **语音驱动设计的背后，是 monorepo 让'意图 → 形态'的距离缩短到 10 秒。**没有 monorepo，语音驱动就是空话；有 monorepo，语音驱动才能稳定落地。"

>
> **这不是工具变革，是组织哲学。**
>
> monorepo 也不只是技术架构。**它是组织哲学——设计师和工程师第一次拥有共同的代码语言。**"
