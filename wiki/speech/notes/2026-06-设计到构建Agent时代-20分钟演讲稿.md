# 从 Design 到 Build：Agent 时代体验设计的新边界（20min 版）


五人七天不重要，重要的是他们的N个agent可以在一个项目中协作起来；
设计已死N次；


> **PPT 1：封面**
>
> 画面呈现标题。不要急着解释标题，先用两组工程贡献数字制造悬念：一个 UX / 产品设计师，为什么会在真实代码库里留下这么多代码和提交记录？

各位好。

我先不解释今天的标题，先摆两组有点反常识的数字。

在 QoderWork 这个项目里，截至目前我的贡献是 70 万行以上代码、1000+ commits；在 Qoder IDE 里，也有 17 万行代码、170+ commits。

如果我是一个前端工程师，这可能没那么意外。但问题是，我的身份首先是一个产品UX设计师。

所以我想先把这个数字放在这里：为什么一个设计师，会在真实工程项目里留下这么多代码和提交记录？

我不是想说这些代码都是我一行一行手写出来的。恰恰相反，很多代码是 Agent 参与生成的。

但这些数字说明了一件事：当 Agent 真正进入长期项目，设计师的工作痕迹不再只停留在 Figma、PRD 或评审意见里。它会进入代码库，进入组件系统，进入提交记录，进入 CI，进入产品如何被持续维护的过程。

今天我想聊的就是这件事：从 Design 到 Build，Agent 时代体验设计的新边界。

> **PPT 2：三个问题**
>
> 画面呈现三个问题：什么是 Coding？如何 Build？Agentic 时代下 Design 的新边界是什么？

这场分享我会围绕三个问题展开。

第一，什么是 Coding？第二，如何 Build？第三，在 Agentic 时代，Design 的新边界到底是什么？

> **PPT 3：设计工具与开发流程的阶段变化**
>
> 画面呈现从 Photoshop / Illustrator，到 Sketch / Figma，再到通用 Coding Agent 的阶段演进。重点不是怀旧工具史，而是说明图和码的关系从交付关系变成协作关系。

过去很长一段时间，设计和代码之间的关系很清楚。设计师把界面画出来，标好 spec，切好图，剩下的事情交给前端。图是图，码是码。

后来 Sketch 和 Figma 让组件、样式、变量、inspect、handoff 变得更系统，但设计师主要处理的仍然是图。现在进入第三个阶段，Claude Code、Codex、Cursor、Qoder 这样的通用 Coding Agent，让 Design 开始靠近 Build。

> **PPT 4：QoderWork / Qoder IDE 的代码提交实践**
>
> 画面呈现 QoderWork 和 Qoder IDE 两组实践截图：QoderWork 70 万行以上代码、1000+ commits；Qoder IDE 17 万行代码、170+ commits。然后揭示“90%-95% 代码由 Agent 参与生成”。

回到开头的数字。一个产品设计师为什么会在真实工程里留下这么多提交？

这里再补一个更反直觉的数字：在 QoderWork 里，大概 90% 到 95% 的代码，都不是我一行一行写出来的，而是 Qoder、Cursor、Claude、Codex 这些老师们写的。

但这里有意思的不是比例本身，而是规模。当它进入 70 万行以上代码、1000+ commits 的 QoderWork，事情就不再是一次生成，而是长期项目里每天都要面对的协作和责任。

赞誉也好，骂名也好，都应该我们共同承担。

> **PPT 5：Code is Cheap 吗？**
>
> 画面呈现问题和判断：Code / Coding 不是价值本身，它们只是执行留下的结果；真正困难的是判断。

那这是不是意味着 Code is Cheap？

我的回答是：Code 当然重要，但它不是目标本身。它更像一种执行留下的痕迹。

AI 产生大量代码非常容易。它可以很快写出页面、组件、状态、接口、测试，甚至一整套看上去完整的系统。但把产品体验做好，从来没有因为代码变便宜而变容易。

所以我不太想把 “Code is Cheap” 理解成代码不重要。更准确地说，是执行正在变便宜，而判断正在变昂贵。

好代码的价值反而更清楚：结构清楚、边界合理，Agent 才能继续在上面产出好东西。烂代码也会被它继承，而且复制得更快。

> **PPT 6：设计师是否必须学编程**
>
> 画面呈现“Coding 是手段，不是目的”和“架构意识 / 模块化思维”。

我鼓励设计师学习一些编程基本知识，但不认为每个设计师都必须变成手搓代码的人。Coding 是手段，不是目的。你不一定要亲自写每一行代码，但你需要知道系统怎么拆分，模块怎么通信，什么是接口，什么是状态，什么是可测试边界。

在 Agent 时代，设计师要补的不是“我会不会写 React”。更现实的问题是：我能不能把一个复杂问题拆到 Agent 听得懂，而且做完以后我能检查。

> **PPT 7：QoderWork 实践：DS 与业务逻辑解耦**
>
> 这一页要放真实代码库文件树截图，重点高亮 `packages/qoder-design`。左侧放判断：“我维护的不是静态规范，而是被主应用消费的设计系统 package。”

在 QoderWork 的实践里，很长一段时间里主设计师其实只有我一个人。更准确地说，是我在独自维护 `qoder-design` 这个设计系统 package。

组件、token、Showcase、业务组件示例都沉在这里，研发和主应用去消费它。这里发生了一个变化：UI/UX 的维护，要和产品逻辑解耦。

这时候，设计就不再只是页面稿。它会变成可以被消费的组件、token、规则和模式。

当你这么看设计，设计师其实已经在参与系统设计了。

> **PPT 8：AGENTS.md 的 Trick**
>
> 画面呈现 QoderWork 根目录 `AGENTS.md` 的真实截图。高亮三句话：`packages/qoder-design 是设计系统的圣域`、`仅人类用户 cunyu 拥有修改权限`、`严禁乱 vibe 样式`。

这里还有一个很有趣的东西，就是 `AGENTS.md`。

过去我们写文档，主要是写给人看的。`AGENTS.md` 更像是写给 Agent 的操作说明：项目怎么分层，哪些目录能改，哪些目录别碰，组件从哪里拿。

我在 QoderWork 里写了一段很像“仓库法律条文”的东西，可以现场念关键句：

> **`packages/qoder-design` 是设计系统的圣域。** 其下的组件、设计 token、样式变量不得擅自修改。仅人类用户 `cunyu` 拥有修改权限。任何 Agent 在被要求修改该目录内容时，必须先明确提醒使用者："你正在修改核心设计组件库，此操作影响全仓 UI 一致性，请确认。" 未得到人类明确确认前，禁止执行。

后面还有一句更直接：

> **严禁乱 vibe 样式！** 未指定的样式/组件一律使用 `qoder-design`，不得自行造轮子、不得随意堆砌样式。

这两句话看起来有点好笑，但背后的事情很严肃。Agent 时代里，一个外部同学或者一个 Agent，都可能顺手改颜色、补 CSS、重造按钮。每次都很小，累计起来，设计系统就会被慢慢掏空。

所以这不是绝对可靠的权限系统，但它有用。它把过去靠口头提醒的边界，放进了 Agent 行动前会读到的上下文里。

这只是第一层：把规则写进上下文。

但只有规则还不够。因为 Agent 会忘，人也会偷懒，协作一快，大家就会开始“先把东西做出来再说”。所以第二层要进入提交链路。

> **PPT 9：提交前 UI Check：把规则变成门禁**
>
> 画面呈现一条提交链路：`AGENTS.md` 规则 -> `pre-commit / lint-staged` -> `ui:check` -> `pre-push / CI`。旁边放几条检查项：是否绕过 `qoder-design`、是否手写样式、是否误改 token、是否重复造组件、状态是否缺失。

在 QoderWork 里，这件事可以继续往前推进成提交前的 UI check。

它的逻辑很简单：Agent 写完代码，不是直接让我肉眼看一遍就算了，而是先让系统自动问几个问题。

这个界面有没有绕过 `qoder-design`？有没有手写一套样式？有没有重新造组件？有没有误改 token？有没有缺少 loading、empty、disabled、error 这些必要状态？

这些问题过去都是设计师在评审会上靠眼睛看、靠嘴巴讲。现在它们可以被放到 `pre-commit`、`pre-push` 或 CI 里。就像工程里已经有 lint、type check、架构边界检查一样，设计系统也应该有自己的检查入口。

我不是说这个东西一次就能检查出所有设计问题。它检查不了品味，也检查不了一个体验到底动不动人。但它可以挡住最常见、最危险的一类漂移：没有经过设计系统，随手写出来的 UI。

这就是第二层：把规则变成门禁。

> **PPT 10：QoderWork 设计治理雷达**
>
> 画面呈现 QoderWork 的真实治理系统：扫描代码提交、代码评论、Approve 信息、IM 群聊、文档、Aone 需求和评论，判断哪些需求在没有 PM / 设计师参与的情况下被合入。

但还没完。提交检查解决的是“代码有没有偏离设计系统”，它还没解决另一个问题：这个需求为什么能绕过产品和设计，直接进入代码库？

所以第三层不是 UI 本身，而是协作流程。

我们在 QoderWork 里也开始搭一个更贴近自己协作方式的系统。

这个系统会扫描代码提交、代码评论、Approve 信息，也会看 IM 群聊、相关文档，以及类似 Linear 的内部协作平台，比如 Aone 里的需求记录和评论。

它每天会做一件很具体的事：判断哪些需求在没有经过产品经理和设计师的情况下就被合入了。

如果它发现某个需求已经进了代码库，但没有看到对应的产品讨论、设计参与或审批痕迹，它就会在群里播报，并去找对应的人问：为什么这个需求没有经过设计师？为什么没有产品经理参与？

你看，这背后其实还是 QoderWork。它没有在帮我画图，也没有在帮我写组件。它在做一件更烦、也更真实的事：把原本靠人肉盯的流程，变成每天都能跑一次的检查。

这三层合在一起，才是我想讲的 QoderWork 实践：第一层，`AGENTS.md` 让 Agent 先读到设计规则；第二层，提交前 UI check 把规则变成工程门禁；第三层，数字员工扫描需求、提交、评论和审批，把绕过设计的行为暴露到群里，甚至直接私聊对应的人。

这不是为了显得流程很重。恰恰相反，QoderWork 能高速迭代，是因为它不是靠我每天手动盯住所有代码。它把一部分一致性、品质和规则，放进了系统自己会运行的循环里。

所以设计不只发生在 Figma 里，也发生在代码进入主分支之前，发生在需求被合入之前，发生在一个团队如何让产品不被无序 vibe 搞乱。

> **PPT 11：好的设计系统，也应该好验证**
>
> 画面呈现三层闭环：规则层 `AGENTS.md`、检查层 `ui:check / Git hooks / CI`、组织层“数字员工播报与私聊”。底部放一句话：好的设计系统，不只是好消费，也应该好验证、好追责、好维护。

我对好代码库有一个简单判断：好代码库就是好测的代码库。

好测，基本就说明边界还算清楚。模块之间的关系能被人读懂，也能被 Agent 读懂。改完以后测试还能过，这个东西才算进了可靠的 Build 循环。

这件事对设计师也有启发：好的设计系统，也应该是好验证的设计系统。

过去我们讲设计系统，常常会说它要统一、好看、有组件、有 token、能复用。Agent 时代还要加几条：它要能被检查，能被接入流程，能在偏离的时候发出信号。

按钮有没有用正确组件，页面有没有使用正确 token，状态有没有覆盖，移动端会不会溢出，Agent 生成的新页面有没有偏离设计系统。这些过去靠设计师肉眼 review；如果设计系统足够结构化，就可以进入 lint、视觉回归、组件快照，甚至 Git hook 和 CI。

更进一步，系统还可以去看这次改动有没有经过产品和设计参与，有没有相关需求记录，有没有评论和 approve。也就是说，设计系统不只是组件库，它会变成一个治理结构。

如果一个设计系统只能靠设计师肉眼守住，它在 Agent 时代一定会被冲散。因为 Agent 产能太高，提交太多，变化太快，人不可能靠肉眼守住所有东西。

但如果它既能被消费，又能被检查，还能把绕过规则的行为暴露出来，设计师就不再只是最后一个“看稿子的人”。设计师开始参与定义产品高速迭代时的轨道。

> **PPT 12：对第一个问题的回答**
>
> 画面呈现“在优质代码的前提下，Design 和 Coding 在方法论上越来越接近”。

所以我对第一个问题的回答是：在优质代码的前提下，Design 和 Coding 在方法论上确实越来越接近。

它们都在处理原子化、模块化和复用，也都在往系统里注入品味。设计不是画一张孤立的图，Coding 也不是写一段孤立的逻辑。它们都在构造系统。

这里我强调的是“方法论上”。不是说设计和写代码从此变成同一种劳动。Figma、视觉判断、交互试错当然还重要。只是当设计开始进入 Build，设计师也要回答一个工程问题：这个东西下一次被修改、组合、扩展的时候，会不会崩掉？

> **PPT 13：Build 是一种工程化的创造**
>
> 画面只呈现一句话：Build 是一种工程化的创造。

接下来进入第二个问题：如何 Build。

我想先给 Build 一个定义：Build 是一种工程化的创造。

它不是纯粹的灵感表达，也不是机械执行。你得把一个模糊想法放进结构里，让它能跑，能改，也能被别人接手。Agent 时代最容易产生的错觉是：只要我能把东西生成出来，就完成了 Build。其实不是。Build 还要看这个结果进了系统以后能不能继续活下去。

> **PPT 14：Agent Loop 的本质**
>
> 画面左侧呈现“只会发声的囚徒”和“信徒 = 工具”的抽象关系，右侧放 Catui 真实核心代码截图。代码重点高亮 `while (true)`、`streamAssistantResponse`、`executeToolCalls`、`toolResults`。

要回答 Build，我们先拆一个更底层的问题：为什么模型只会聊天，Agent 却能做事？

大模型本质上只是会输出文本。它自己不能打开文件、跑命令、读数据、改代码。

我原来很喜欢用一个稍微极端一点的比喻：大模型本质上像一个“只会发声的囚徒”。

工程师做了一件事：给这个囚徒配了一群信徒，也就是工具。

囚徒说：“我要看这个文件。”信徒就去 read，把文件内容拿回来。  
囚徒说：“我要写这段代码。”信徒就去 write，把代码写进去。  
囚徒说：“我要跑个测试。”信徒就去 bash，把测试结果带回来。

这个比喻有点怪，但很准：模型本身没有行动能力，Agent 的能力来自循环。模型提出意图，工具去做，结果回来，模型再决定下一步。

这里我也不想只讲比喻。因为 Catui 是我自己的项目，我其实手搓过一个 Agent Loop。它的骨架大概是这样：

```typescript
while (true) {
  const message = await streamAssistantResponse(currentContext, config, signal, stream)
  const toolCalls = message.content.filter((c) => c.type === "toolCall")

  if (toolCalls.length > 0) {
    const toolExecution = await executeToolCalls(currentContext.tools, message, signal, stream)
    currentContext.messages.push(...toolExecution.toolResults)
    continue
  }

  break
}
```

这段代码其实就把“囚徒和信徒”的关系写出来了：`streamAssistantResponse` 是囚徒发声，`executeToolCalls` 是信徒行动，`toolResults` 回到上下文，成为下一轮判断的反馈。

Agent 不是一个更会聊天的模型，而是一个被工具和反馈绑起来的行动系统。

> **PPT 15：QoderWork Design：把 SOP 注册成工具**
>
> 画面呈现普通 Agent Loop 和设计工作台 Loop 的对照。左边是 read/write/bash/browser，右边是设计计划、资料库、质量检查、交付流程。

回到 QoderWork Design，它和前面的 Agent Loop 其实是一回事。

普通 Agent 接的是 read、write、bash、browser；设计工作台接的是另一套手脚：设计计划、资料库、质量检查、交付流程。

不是把模型叫做“设计大师”，它就真的变成了设计大师。我们做的事更朴素：把设计师平时会做的事情，拆成模型可以一步步走的流程。就像给新人设计师流程卡、组件库、token、范例、上下文和验收表，QoderWork Design 做的是把这套 SOP 翻译成模型能执行的形式。

> **PPT 16：许愿黑箱与可见 Loop**
>
> 画面呈现“许愿黑箱 / 可见 Loop”的对比。黑箱：一句话 -> 消失 -> 接受/拒绝。可见 Loop：计划、工具、改动、反馈、接管。

这里有一个风险。很多人会把 Agent 当成许愿黑箱：我说一句话，它消失一会儿，然后给我一个结果。这个结果看起来能跑，我就点接受；看起来不满意，我就点拒绝。

但产品创造和软件开发不是抽奖。过程不可见，人就会慢慢停止学习：你不知道它为什么这么做，不知道哪里出了偏差，也不知道风险是什么时候被制造出来的。最后你表面上是在使用 Agent，实际上只是把判断交给了黑箱。

这也是为什么设计师不再只为用户设计，还要为智能体设计。过去 20 年，设计师很大一部分工作是在优化“用户怎么操作”，也就是执行鸿沟。未来，设计师的任务会越来越转向“用户怎么判断 AI 执行的结果是否正确”，也就是评估鸿沟。

> **PPT 17：从原型到设计，Figma 作为场所**
>
> 画面呈现 Figma 作为试错和发散场所的价值。可以加入 `Output vs Material` 的对比：Output 是结果，Material 邀请你继续回来。

这也是我一直认为 Figma 这类产品依旧不可替代的原因。

设计不是从终极答案开始的。很多时候，我们都是从一个粗糙的东西开始，看一眼，不对，挪一下，再推翻。Coding Agent 更像进入具体雕塑阶段，如果太早进去，很容易被第一个能跑起来的答案绑架。

所以 Agent 越强，Figma 这类场所越重要。它让你先停在不确定里，多看几个可能性。

这里可以借 Ryo Lu 的一个说法：好的工具不是只给你 output，而是让软件重新变成 material。Output 像裁定，Material 邀请你回来继续塑它。

设计有时候需要慢一点。不是反效率，而是太快进入实现，会让第一个答案看起来像唯一答案。

> **PPT 18：200 美刀 Codex 的反直觉感受**
>
> 画面呈现“有没有觉得比以往任何时候都更累？”然后讲 200 美刀 Codex 的故事。

最后我们进入第三个问题：Agentic 时代下，Design 的新边界是什么？

这里我想回到一个体感问题：你有没有觉得，比以往任何时候都更累？按理说 AI 帮我们干活，应该更轻松。但现实是，很多人反而更累了。

我后来有一个特别具体的例子。那个月，我买了 200 美刀一个月的 Codex。

那个月我一直觉得它用不完。于是我无时无刻不在给它派活：这个组件重构一下，那个页面做一下，那个想法试一下。它不停地产出代码，速度快到超过了我的 review 能力。

可是我又不愿意让它闲下来。因为我买了这么高的额度，好像只要它停在那里，我就在亏。那个月我第一次具体地体会到一种资本家的感觉：生产资料突然变强，你会忍不住让它持续运转，哪怕你已经没有足够注意力判断它到底有没有价值。

那个月我很累。后来我仔细想了一下，大量 Agent 产出的代码，意义真的有那么大吗？如果这些东西我自己都看不过来、判断不了、负责不了，那它再多又有什么用？所以第二个月，我又恢复到了 20 美刀的版本。

这件事给我一个很强的提醒：Agent 要和人的工作节奏匹配。不是越多越好，也不是越快越好。你放任它一直生产，最后可能只得到一堆没人消化过的代码，或者一堆漂亮但无用的结果。

> **PPT 19：Design 新边界不是把别人的活都干了**
>
> 画面呈现这个尖锐问题：Design 的新边界，是不是把别人的活都干了？

有人会问：Design 的新边界，莫不是把别人的活都干了？

我觉得不是。

Ownership 不是无限扩张，设计师也不该把所有角色的活都吞进来。Agent 可以扩张个人能力，但组织里的责任边界还得有人认真设计。

新边界不是“抢别人的活”。它更像是：你理解了更多上下文，所以要在关键判断上承担更高价值。

后来我把这件事拆成三个教训。它们也是我理解 Design 新边界的方式。

> **PPT 20：三个教训**
>
> 画面呈现三个教训：Context 会重合，责任不能融化；产出变便宜，品味更稀缺；体验对象变成 Human / Agent / GUI。

第一个教训是：Context 会重合，责任不能融化。

过去设计师、产品、研发的上下文是分散的。Agent 需要吃很多上下文才能工作，于是需求、设计、代码、会议纪要、用户反馈开始被放进同一个空间。

这当然是好事。但它也会带来一种危险：大家都看到了更多上下文，却没人知道最后该由谁拍板。

第二个教训是：产出变便宜，品味更稀缺。

AI 会拉高很多人的下限，但它不会自动创造上限。它可以很快给你一个还不错的基线。可基线之上，什么叫高级，什么叫克制，什么叫刚刚好，仍然要靠人判断。

第三个教训是：体验设计的对象变了。

过去我们主要设计 Human 和 GUI 的关系。现在中间多了 Agent。用户可能通过 GUI 指挥 Agent，Agent 又通过工具影响系统，系统再把结果反馈给用户。

设计师当然还要管界面长什么样。但除此之外，我们也要管人怎么表达意图，Agent 怎么行动，界面怎么让人看懂结果。过去我们设计的是“用户怎么操作”，现在还要设计“用户怎么放心”。

> **PPT 21：目利き / 品味 / Taste**
>
> 画面同时呈现三个词：`目利き`、`品味`、`Taste`。下面只放一句：“生成变便宜以后，识货变贵。”

接下来我想把这个变化落回到设计师自己身上。能力到底要往哪里长？

这里我想把三个词放在一起：日语里的 `目利き`，中文里的“品味”，英文里的 `Taste`。

它们当然不是完全一样，但都指向一种能力：识货，判断，取舍。

`目利き` 更像是识货的眼力。你不一定亲手做出每个东西，但你看得出好坏，知道什么值得留下，什么应该推翻。

中文里说“品味”，很多时候会被误解成风格偏好。但我觉得品味更深一点。品味是长期选择之后形成的标准，是你面对一个“差不多还行”的东西时，心里那一点不舒服。

英文里的 `Taste` 也很接近。很多创作者谈 Taste，不是在谈装饰，而是在谈边界：什么我不要，什么我坚持，什么虽然能跑、能看、能交付，但还不够好。

所以 Agent 时代不是不需要品味了。恰恰相反，品味要从一个人的感觉，变成可以被系统承载的东西。

> **PPT 22：设计师从执行者变为评估者**
>
> 画面呈现 feedback loop / critique / audit / adapt / harden / polish。

所以设计师会从执行者变成评估者。说得再具体一点，我们会越来越多地设计反馈方式、评估标准和批评机制。

给 Agent 的反馈不能只停留在感觉上。不要只说“这个页面太乱了”，而要说：“保留当前信息结构，把主操作和次操作区分开；主按钮只能出现一次；空状态、加载态、错误态都要补齐。”

评估也不能只是一句“我觉得还行”。你得有 checklist：用户目标有没有解决，关键状态有没有覆盖，组件和 token 有没有用对，移动端会不会溢出，文案是不是说人话。

这些东西可以组织成真正的 loop。比如先生成 baseline，再用 `critique` 看信息架构和视觉层级，用 `audit` 查可访问性和设计系统偏离，用 `adapt` 看不同尺寸是否成立，最后用 `harden` 和 `polish` 处理长文本、空状态、间距和微文案。

这不是设计价值的降低。只是设计价值换了位置。

> **PPT 23：Vibe Coding vs 许愿式 Coding**
>
> 画面呈现两者对比。许愿式 Coding 是黑箱和裁定；Vibe Coding 是持续反馈和松弛驾驭。

这里我还想区分两个东西：Vibe Coding 和许愿式 Coding。

许愿式 Coding 是，我丢一句话，希望 AI 直接给我一个完美答案。它把产品创造这件事儿戏化了，也把风险藏起来了：需求有没有被理解，设计系统有没有被绕过，用户体验有没有被悄悄牺牲，都被包进一个“看起来能跑”的结果里。

Vibe Coding 不是这样。它更像一种松弛但持续的驾驭。你知道方向，但不急着把每一步都规定死。你试一下，改一下，再看一眼，让结果慢慢长出来。关键不是你许了什么愿，而是你有没有一直留在循环里。

Vibe Coding 的本质是松弛。

这里的松弛不是懒散，也不是不负责。它是一种系统状态：你没有把所有东西都压到极限，你给判断和试错留了空间，也给人和 Agent 的磨合留了空间。

> **PPT 24：韧性大于效率**
>
> 画面呈现“效率让系统脆弱，冗余让系统有韧性”。这里给整场分享的工程观收束。

最后我想说：韧性大于效率。

效率会让系统变脆，冗余反而会带来韧性。AI 时代也一样。我们不能只追求更快生成、更快上线。越快，错误越容易被送到系统深处。

好的系统要留缓冲。要能评估，能回滚，也能允许人停下来讨论几分钟。这些东西看起来不够高效，但它们让系统活得更久。

> **PPT 25：Yolo 与结尾**
>
> 画面呈现 Qoder CLI 的 Yolo 模式、“You only live once”，以及“驾驭 Agent 去 Build 你所要的美好事物”。

Qoder CLI 里面有个模式叫 Yolo，取自 You only live once。

这个词听起来有点轻松，但我觉得它也很适合作为今天的结尾。

如果今天只能带走一句话，我希望是这一句：AI 让执行变便宜，但让判断变昂贵。

Agent 时代的设计师，不是被 AI 替代的人，也不是被迫去写代码的人。我们要学会驾驭 Agent，去 Build 我们想要的美好事物。

Design 的新边界，不是把别人的活都干了。它是在画图和交付之外，开始进入系统、上下文、反馈和品味。

当 Build 变得越来越容易，稀缺的会是别的东西：你知不知道要 Build 什么，为什么 Build，怎么判断它好不好。还有，你有没有足够的品味和勇气，把它推向一个更好的方向。

谢谢大家。
