## 摘要

Claude Code 的核心，其实就是一个 while 循环：模型生成响应，如果里面有工具调用，就执行工具，把结果塞回上下文，再让模型生成下一步，持续往复。
	
就这么一个循环，被工程化成了完整产品，代码规模 30 多万行。
	
从设计上看，Claude Code = 模型 + Harness。Harness 又可以拆成三件事：工具系统、上下文工程、自主循环。
工具系统里，CC 做了很多标准化约束。它不会鼓励模型直接执行 find、grep、cat、head 这类通用行为，而是引导模型走 GrepTool、GlobTool 等专用工具。原因是内建工具可以输出可审计、结构化的日志，让每一步行为更透明，也更容易被约束。
	
工具本身也有权限级别和校验逻辑。比如 Edit 为了避免交叉覆盖，会要求先 Read；Git 遇到 push force 这类高风险行为，会有 prompt 约束和 UI 警告。核心目的不是让模型更自由，而是给模型画边界，让错误不要扩散。
	
上下文管理更夸张。CC 会通过 MicroCompact、AutoCompact、会话压缩、记忆替换和裁剪策略，尽量保证模型只看到当前任务真正相关的信息。
	
文件加载也用了 Just-In-Time 思路：工具定义和能力不一次性全塞进去，只保留路径，需要时再读。Sub-Agent 则通过上下文隔离，把不同子任务的信息分开，降低主循环的认知负载。
	
更有意思的是，模型本身也开始为了 Harness 变得更适合 Agent。比如新版本 Opus 在指令遵循上更强调“按字面执行指令”。这对 Agent 很关键，因为行为边界往往就写在 system prompt 里。
	
OpenClaw、Hermes Agent、Claude Code 这类系统产生的大量工具调用数据，也会继续反哺模型。未来模型一定会逐步内化工具调用、上下文压缩，甚至自我约束边界。
	
所以今天 Claude Code 里写的这些 Harness 逻辑，长期看也会被模型吃掉。Harness 很强，但它可能也是一个过渡产物。🐶
